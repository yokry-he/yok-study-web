# 阶段任务清单

## 适合谁看

适合已经选定学习方向，但不知道每天、每周应该做什么的人。

学习路线不能只停留在“先学 A，再学 B”。真正有效的路线应该能回答：

- 这一阶段要解决什么问题。
- 应该读哪些文档。
- 要做出什么可运行结果。
- 怎么判断可以进入下一阶段。
- 常见卡点怎么处理。

## 使用方式

每个阶段都按同一套节奏推进：

```text
读核心文档
↓
做最小练习
↓
完成阶段任务
↓
记录问题
↓
做能力自测
↓
进入下一阶段
```

不要只读不做，也不要只复制代码。每个阶段都要留下一个可以运行、可以解释、可以复盘的结果。

## 阶段 1：前端基础

目标：能写出结构清晰、样式稳定、移动端可读的页面。

建议学习：

- [前端基础](/frontend/html-css)
- [CSS 学习导览](/css/introduction)
- [JavaScript 基础](/javascript/fundamentals)
- [JavaScript 真实项目问题库](/projects/issues-javascript)
- [浏览器学习导览](/browser/introduction)

任务：

1. 写一个个人资料页。
2. 写一个列表页，包含筛选栏和卡片或表格。
3. 做移动端适配。
4. 给按钮、链接、表单加上基础交互。
5. 做一次 JavaScript 列表交互和排障加练。
6. 用浏览器 DevTools 检查布局、网络请求、Storage 和控制台错误。

验收：

- 页面语义清晰。
- 移动端没有横向溢出。
- CSS 不依赖宽泛选择器。
- 能解释一次点击事件如何触发 JS 逻辑。
- 能复现并修复至少 2 个 JavaScript 常见问题。
- 能说明浏览器如何加载 HTML、CSS、JS。

常见卡点：

- Flex 和 Grid 混用混乱。
- 只会写固定宽度。
- JS 变量和作用域不清楚。
- 不会看控制台错误。

## 阶段 2：Vue 前端项目

目标：能用 Vue 3 完成一个有路由、状态、请求、表单和权限的中小型前端项目。

建议学习：

- [Vue 学习导览](/vue/introduction)
- [Vue Router](/vue/router)
- [Pinia 状态管理](/vue/pinia)
- [请求与接口封装](/vue/request)
- [权限与菜单](/vue/permission)
- [前端工程化学习导览](/engineering/introduction)

任务：

1. 创建 Vue 3 + TypeScript 项目。
2. 建立页面路由和布局。
3. 实现登录页和用户信息状态。
4. 封装请求层和错误处理。
5. 实现用户列表、筛选、分页和编辑表单。
6. 实现动态菜单和按钮权限。
7. 写 README，说明启动、配置和目录结构。

验收：

- 页面、状态、请求、权限分层清楚。
- 路由守卫能处理未登录和无权限。
- 请求错误有统一提示和兜底。
- 表单有校验和提交状态。
- 构建通过，关键页面移动端可用。

常见卡点：

- 把 API 请求写在组件各处。
- Pinia store 变成所有逻辑的大杂烩。
- 权限只做前端隐藏，没有服务端校验意识。
- 环境变量和接口代理混乱。

## 阶段 3：Node API 与数据库

目标：能写出可维护的后端 API，并连接数据库完成真实业务数据流。

建议学习：

- [Node.js 学习导览](/node/introduction)
- [HTTP API 开发](/node/http-api)
- [鉴权与会话](/node/auth-session)
- [数据库学习导览](/database/introduction)
- [数据建模与表设计](/database/modeling)
- [事务、锁与并发](/database/transactions)

任务：

1. 创建 Node API 服务。
2. 实现 `/health` 健康检查。
3. 实现登录、退出和当前用户接口。
4. 设计用户、角色、权限表。
5. 实现用户 CRUD。
6. 实现角色授权。
7. 增加统一错误响应和结构化日志。
8. 写数据库迁移说明和字段注释。

验收：

- controller、service、repository 分层清楚。
- 参数校验、鉴权、授权都在服务端执行。
- 数据库字段、约束、索引有业务说明。
- 事务只包住必须原子完成的数据库操作。
- 关键错误能通过日志定位。

常见卡点：

- 直接在路由里写 SQL。
- 数据库只建字段，没有约束和注释。
- 事务里混入远程请求。
- 登录态和权限边界不清楚。

## 阶段 4：部署、观测与排错

目标：能把项目部署到可访问环境，并具备基础回滚和问题排查能力。

建议学习：

- [前端构建与部署](/engineering/build-deploy)
- [DevOps 学习导览](/devops/introduction)
- [Nginx 静态部署与代理](/devops/nginx)
- [Docker 容器化](/devops/docker)
- [CI/CD 自动化发布](/devops/ci-cd)
- [可观测性](/devops/observability)

任务：

1. 构建前端生产产物。
2. 用 Nginx 部署静态资源并代理 API。
3. 用 Docker Compose 启动前端、后端和数据库。
4. 配置不同环境变量。
5. 写 CI 流程，至少跑检查和构建。
6. 写发布检查清单和回滚步骤。
7. 增加 request id、错误日志和健康检查。

验收：

- 二级路由刷新不 404。
- API 代理路径正确。
- 生产环境不暴露私密变量。
- 有版本号、发布记录和回滚方式。
- 上线后能通过日志和健康检查定位问题。

常见卡点：

- 本地代理当成生产代理。
- `index.html` 被强缓存。
- 环境变量构建时和运行时混淆。
- 只会发布，不会回滚。

## 阶段 5：AI 工程接入

目标：能把模型能力接入业务，而不是只做一个演示页面。

建议学习：

- [AI 工程学习导览](/ai-engineering/introduction)
- [LLM API 调用](/ai-engineering/llm-api)
- [提示词工程](/ai-engineering/prompt-engineering)
- [RAG 检索增强生成](/ai-engineering/rag)
- [评测与质量保障](/ai-engineering/evaluation)

任务：

1. 封装模型调用服务。
2. 建立 prompt 模板和版本。
3. 做一个文档问答或工单分类功能。
4. 增加输入校验和输出结构校验。
5. 记录调用日志、耗时和成本。
6. 建立 20 到 50 条评测样例。
7. 处理拒答、权限过滤和人工兜底。

验收：

- 模型调用不直接散落在组件里。
- prompt 可版本化和回滚。
- RAG 答案能给出来源。
- 权限由业务系统判断，不交给模型决定。
- 评测能发现 prompt 或模型变更带来的退化。

常见卡点：

- Demo 好用就直接上线。
- 没有评测集。
- 检索结果差却只改 prompt。
- 成本和延迟没有监控。

## 每阶段复盘模板

完成一个阶段后，建议记录：

```md
## 阶段名称

### 本阶段完成了什么

### 最难的问题是什么

### 我是怎么解决的

### 哪些知识还不稳

### 下阶段要重点注意什么
```

复盘比单纯收藏资料更重要。它能把“学过”变成“能复用的经验”。
