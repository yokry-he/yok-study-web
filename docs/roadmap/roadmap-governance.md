# 路线维护规则

## 这个页面解决什么

学习路线会随着技术模块扩展而变化。如果没有维护规则，路线很容易变成过时链接、重复章节和无法执行的知识清单。

这个页面定义路线模块的治理方式：

- 什么内容应该进入路线。
- 路线如何和技术模块关联。
- 新增模块后如何更新路线。
- 如何判断路线已经过时。
- 如何保持路线可执行。

## 路线的组织原则

本站路线采用任务优先，而不是技术百科优先。

也就是说，路线不是这样组织：

```text
HTML
CSS
JavaScript
Vue
Node
Database
```

而是这样组织：

```text
我要能写页面
↓
我要能做项目
↓
我要能写接口
↓
我要能接数据库
↓
我要能部署上线
↓
我要能排查问题
```

技术模块负责讲清知识，路线负责告诉读者什么时候学、学到什么程度、做什么项目验证。

## 新增路线内容的标准

新增路线页面前先问：

1. 它是否对应明确目标人群。
2. 它是否能给出阶段顺序。
3. 它是否有阶段任务。
4. 它是否有验收标准。
5. 它是否链接到已有技术模块。
6. 它是否避免和已有路线重复。

如果只是“某技术学习清单”，优先放到技术模块，而不是路线模块。

## 路线页面模板

每条路线建议包含：

```md
# 路线名称

## 适合谁看

## 最终目标

## 学习路径

## 阶段任务

## 阶段验收

## 推荐项目

## 常见误区

## 下一步学习
```

路线页面必须指向具体文档，不要只写泛泛建议。

## 路线和模块状态同步

当某个模块从 first-version 升级为 stable 时，要检查：

- 路线页面是否引用了新的核心章节。
- 阶段任务是否需要补充。
- 项目里程碑是否需要更新。
- 能力自测是否需要新增能力项。
- 技术库总览是否需要更新描述。
- 模块状态表是否需要更新文档数和下一步。

例如 Node.js 模块补齐鉴权、数据库、测试、安全后，Node 后端路线也应该覆盖这些能力，而不是还停留在 HTTP API 和部署。

## 路线链接规则

路线链接应该优先指向：

1. 模块导览页。
2. 关键核心章节。
3. 项目实战页。
4. 问题库页。
5. 速查手册。

不要把路线直接链接到太深的边缘章节，除非该章节是阶段任务必须阅读的内容。

## 过时检查

每次大规模扩展后，检查路线是否有这些问题：

- 链接到不存在的页面。
- 阶段顺序和模块现状不一致。
- 推荐项目和当前技术栈不一致。
- 自测项缺少新能力。
- 路线只讲学习，不讲产出。
- 阶段验收无法验证。

可以用 [模块状态](/contribute/module-status) 作为检查入口。

## 路线版本记录

路线发生明显变化时，建议记录：

```md
## 变更日期

### 变更内容

### 为什么调整

### 影响哪些路线

### 需要同步更新哪些模块
```

路线不是一次写完的。随着站点模块成熟，路线要持续吸收新的稳定章节。

## 内容深度规则

路线页面只写学习顺序、任务和验收，不展开讲完整技术细节。

详细解释放到对应模块：

- Vue 细节放到 Vue 模块。
- Node 细节放到 Node 模块。
- 数据库细节放到数据库模块。
- 部署细节放到 DevOps 模块。
- AI 细节放到 AI 工程模块。

这样可以避免路线页面越来越长，也避免同一知识点多处维护。

## 实际维护场景

### 场景 1：新增了一个技术模块

处理：

1. 在技术库总览增加模块入口。
2. 在模块状态表登记。
3. 判断是否需要加入某条路线。
4. 如加入路线，补阶段任务和自测项。
5. 更新扩展计划。

### 场景 2：某模块变成 stable

处理：

1. 更新路线中对应阶段描述。
2. 增加关键章节链接。
3. 更新项目里程碑。
4. 更新能力自测。
5. 更新模块状态下一步。

### 场景 3：发现路线太长

处理：

- 保留主路线。
- 把细节拆到阶段任务。
- 把技术解释移到模块。
- 把项目要求移到项目里程碑。
- 把检测项移到能力自测。

## 最佳实践

- 路线按目标组织，技术模块按知识组织。
- 每条路线必须有产出和验收。
- 新模块稳定后要同步路线。
- 路线不重复技术细节，只做导航和任务编排。
- 自测项必须能用项目证据证明。
- 模块状态表是路线维护的入口。
