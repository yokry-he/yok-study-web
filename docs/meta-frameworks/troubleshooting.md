# Nuxt / Next 常见问题

## 适合谁看

适合在 Nuxt 或 Next 项目中遇到白屏、hydration mismatch、数据不更新、环境变量异常、部署后 404、服务端和客户端行为不一致等问题的学习者。

这类问题通常不是单个组件写错，而是渲染环境、数据获取位置、缓存策略和部署形态没有理清楚。

## 排查顺序

先按这个顺序定位：

```text
问题发生在本地还是线上
↓
发生在服务端渲染还是客户端交互
↓
页面是否使用缓存或静态生成
↓
数据在哪里获取
↓
运行环境是 Node、静态、Serverless 还是 Edge
↓
最近是否发布或改过环境变量
```

不要一上来就重写组件。先确认问题属于渲染、数据、缓存、环境还是部署。

## 问题 1：hydration mismatch

### 问题现象

- 控制台提示 hydration 不匹配。
- 服务端看到的内容和客户端接管后的内容不一致。
- 页面刚打开正常，随后闪一下变成另一个内容。

### 常见原因

- 服务端和客户端生成了不同随机数。
- 服务端和客户端时间、时区不同。
- 首屏直接读取 `window`、`localStorage`、屏幕宽度。
- 用户态只在客户端可见，服务端渲染的是另一种状态。
- 组件库依赖浏览器环境。

### 解决方案

随机值、当前时间、浏览器宽度等只在客户端生成，或者在服务端和客户端使用同一份初始数据。

浏览器 API 放到客户端生命周期或客户端组件边界内。

用户登录态要明确：

- 是否能在服务端通过 cookie 读取。
- 是否只存在客户端。
- 是否允许服务端渲染这个页面。

### 预防方式

- 首屏渲染不要依赖不稳定值。
- 浏览器 API 不要在服务端执行。
- 客户端专用组件要有明确边界。
- 登录后页面谨慎使用 SSR 和公共缓存。

## 问题 2：页面数据不更新

### 问题现象

- 后台已经改了数据，页面仍显示旧内容。
- 本地开发正常，线上一直旧数据。
- 刷新后还是旧数据。

### 常见原因

- 页面被静态生成。
- 框架 fetch 或页面缓存生效。
- CDN 缓存了 HTML 或 API 响应。
- 服务端缓存或 Redis 仍有旧数据。

### 解决方案

先判断缓存在哪一层：

```text
浏览器 Network
CDN 响应头
框架缓存配置
API Cache-Control
服务端缓存
数据库数据
```

对用户数据、权限数据、后台列表等动态数据，不要使用公共长缓存。

对内容站数据，明确 revalidate 或重新生成策略。

### 预防方式

- 每个页面定义缓存策略。
- 发布文档写清 CDN 清理对象。
- 用户操作后明确刷新或重新验证数据。
- API 响应头不要模糊。

## 问题 3：环境变量在服务端有值，客户端没有值

### 问题现象

- 服务端日志里变量存在。
- 浏览器里读取是 `undefined`。
- 本地正常，生产不正常。

### 常见原因

元框架通常区分私有变量和公开变量。私有变量只能在服务端读取，客户端不能直接访问。

### 解决方案

先确认变量应该给谁用：

| 变量 | 应该在哪读 |
| --- | --- |
| 数据库连接 | 服务端 |
| 私密 API key | 服务端 |
| JWT secret | 服务端 |
| 公开 API 前缀 | 客户端可公开 |
| 站点名称 | 客户端可公开 |

如果客户端需要某个公开配置，按框架规则放到 public runtime config 或公开环境变量中。

### 预防方式

- 不把密钥暴露给客户端。
- 环境变量集中读取。
- 启动时校验必填变量。
- README 写清变量作用域。

## 问题 4：部署后刷新路由 404

### 问题现象

- 首页能打开。
- 从页面内跳转正常。
- 刷新某个路由后 404。

### 常见原因

- 静态托管没有配置 fallback。
- 子路径部署 base 不一致。
- 动态路由没有被预生成。
- 部署方式和项目渲染方式不匹配。

### 解决方案

静态站点要确认 fallback 或预生成策略。

子路径部署同时检查：

- 框架 base。
- router base。
- Nginx 或平台路由配置。
- 静态资源路径。

动态路由如果是静态导出，需要确认构建时能枚举所有路径。

### 预防方式

- 上线前验证二级路由刷新。
- 复制详情页链接到新标签页验证。
- 子路径部署写入 README。
- 不确定动态路由时不要盲目静态导出。

## 问题 5：接口在本地可用，线上不可用

### 问题现象

- 本地 server api 或 route handler 正常。
- 静态部署后接口 404。
- Serverless 环境里某些 Node API 报错。

### 常见原因

- 部署目标不支持服务端运行时。
- 静态导出不包含接口能力。
- Edge runtime 不支持某些 Node API。
- 环境变量没有注入到目标平台。

### 解决方案

先确认部署方式是否支持当前能力：

```text
静态托管：通常没有服务端接口
Node server：支持完整 Node 能力
Serverless：支持函数形态，但有平台限制
Edge：低延迟，但 Node API 受限
```

如果项目需要服务端接口，不要部署成纯静态站点。

### 预防方式

- 选部署方式前列出项目使用的框架能力。
- 对每个环境提供 `/health` 或关键路由检查。
- 部署文档写清运行时限制。

## 问题 6：组件库在服务端渲染时报错

### 问题现象

- 组件库在普通 SPA 正常。
- 在 Nuxt 或 Next SSR 中报 `window is not defined`。
- 页面构建或服务端渲染失败。

### 常见原因

组件库或某个依赖直接访问浏览器 API，不适合服务端执行。

### 解决方案

- 将组件放到客户端专用边界。
- 延迟到客户端加载。
- 选择支持 SSR 的组件库。
- 检查是否有业务代码在顶层访问 `window`。

### 预防方式

- 引入组件库前确认 SSR 支持。
- 浏览器 API 放到客户端生命周期里。
- 用最小页面验证组件是否能 SSR。
