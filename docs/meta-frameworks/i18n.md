# 国际化与多语言站点

## 适合谁看

适合准备做中英文站点、出海产品、国际官网、多语言文档或多地区门户的人：

- 只把文案放进 JSON，但路由和 SEO 没处理。
- 语言切换后页面跳回首页。
- 服务端渲染和客户端语言不一致。
- 多语言内容、日期、货币和表单校验混在一起。
- 不知道 locale 应该放路径、域名还是 Cookie。

国际化不是简单翻译文案。完整多语言站点要同时处理路由、内容、SEO、格式化、缓存、用户偏好和回退策略。

## i18n 和 l10n

| 概念 | 含义 |
| --- | --- |
| i18n 国际化 | 让系统具备支持多语言和多地区的能力 |
| l10n 本地化 | 针对具体语言和地区提供翻译、格式和内容 |

例如：

- i18n：代码支持 `en`、`zh-CN`、`ja`。
- l10n：中文页面使用中文标题、人民币、北京时间和本地案例。

## locale 放在哪里

常见方案：

| 方案 | 示例 | 适合 |
| --- | --- | --- |
| 路径前缀 | `/en/docs`、`/zh/docs` | 内容站、SEO、多语言文档 |
| 子域名 | `en.example.com` | 强地区站点 |
| 顶级域名 | `example.de` | 本地品牌和合规 |
| Cookie / 用户设置 | 用户后台语言偏好 | 登录态产品 |

公开内容站优先使用路径或域名表达语言，这样搜索引擎和分享链接更清楚。

## 翻译内容放哪里

小型项目：

```text
locales/
├─ en.json
└─ zh-CN.json
```

内容站：

```text
content/
├─ en/
│  └─ docs/
└─ zh-CN/
   └─ docs/
```

复杂项目可以使用 CMS，把每篇内容的语言版本、状态、slug、SEO 字段都纳入内容模型。

## Nuxt 国际化

Nuxt 常用 `@nuxtjs/i18n`。

配置思路：

```ts
export default defineNuxtConfig({
  modules: ['@nuxtjs/i18n'],
  i18n: {
    locales: [
      { code: 'en', language: 'en-US', name: 'English' },
      { code: 'zh', language: 'zh-CN', name: '中文' }
    ],
    defaultLocale: 'zh',
    strategy: 'prefix_except_default'
  }
})
```

页面切换语言时，要尽量保持当前页面语义，而不是一律跳首页。

```vue
<script setup lang="ts">
const switchLocalePath = useSwitchLocalePath()
</script>

<template>
  <NuxtLink :to="switchLocalePath('en')">English</NuxtLink>
</template>
```

多语言 SEO 可以结合 i18n 模块提供的 locale head 能力生成语言相关标签。

## Next 国际化

Next App Router 推荐用路由段表达语言：

```text
app/
└─ [lang]/
   ├─ layout.tsx
   └─ page.tsx
```

在 layout 中读取语言：

```tsx
export default async function LocaleLayout({
  children,
  params
}: {
  children: React.ReactNode
  params: Promise<{ lang: string }>
}) {
  const { lang } = await params

  return (
    <html lang={lang}>
      <body>{children}</body>
    </html>
  )
}
```

翻译字典可以按语言加载：

```ts
const dictionaries = {
  en: () => import('./dictionaries/en.json').then((module) => module.default),
  zh: () => import('./dictionaries/zh.json').then((module) => module.default)
}
```

如果项目更复杂，可以使用 next-intl 等生态库，但仍然要理解路由、metadata、缓存和服务端渲染边界。

## 格式化不要手写

日期、数字、货币应使用 Intl API 或成熟库。

```ts
const price = new Intl.NumberFormat('zh-CN', {
  style: 'currency',
  currency: 'CNY'
}).format(199)
```

不要自己拼：

```ts
`￥${price}`
```

不同地区的货币、小数、日期顺序、星期起点都可能不同。

## 多语言 SEO

每个语言版本都需要：

- 本语言 title。
- 本语言 description。
- 正确 `html lang`。
- hreflang。
- canonical。
- 多语言 sitemap。

例子：

```text
/zh/vue
/en/vue
```

两个页面内容对应，但 title、description、正文和结构化数据都应该使用对应语言。

## 缓存边界

多语言站点缓存 key 至少要包含 locale。

危险场景：

```text
用户访问 /en/docs
↓
服务端缓存英文内容
↓
中文用户访问 /zh/docs
↓
拿到英文缓存
```

处理方式：

- URL 中包含 locale。
- 缓存 key 包含 locale。
- API 返回内容按 locale 区分。
- CDN 路径和语言一致。

## 实际项目问题

### 1. 切换语言后跳到首页

**原因**

语言切换只改了 locale，没有计算当前页面对应路径。

**解决方案**

- 为内容建立跨语言关联。
- 使用当前页面 slug 映射目标语言 slug。
- 找不到翻译时提示或回退，而不是静默跳首页。

### 2. SSR 语言和客户端语言不一致

**原因**

服务端根据路径渲染中文，客户端又根据 localStorage 切成英文。

**解决方案**

- 公开页面以路径 locale 为准。
- 用户偏好只用于进入站点时选择默认语言。
- SSR 和客户端使用同一个 locale 来源。

### 3. 多语言页面 SEO 重复

**原因**

不同语言页面使用相同 title、description 和 canonical。

**解决方案**

- 每个语言版本生成独立 metadata。
- canonical 指向当前语言标准 URL。
- hreflang 指向其他语言版本。

### 4. 翻译 key 越来越乱

**原因**

所有文案都放在一个大 JSON。

**解决方案**

- 按页面或模块拆分字典。
- key 使用业务语义，不使用完整中文当 key。
- 删除页面时同步清理翻译。
- 缺失翻译在 CI 或构建时检查。

## 最佳实践

- 公开内容站优先用路径或域名表达 locale。
- SSR 和客户端使用同一个 locale 来源。
- 多语言内容、SEO 和路由一起设计。
- 缓存 key 包含 locale。
- 日期、数字、货币使用 Intl。
- 翻译字典按模块拆分。
- 缺失翻译要有检查机制。
