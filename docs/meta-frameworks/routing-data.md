# 路由、布局与数据获取

## 适合谁看

适合已经看过 Nuxt 或 Next 入门，但对文件路由、布局嵌套、服务端数据获取、客户端请求、缓存和加载状态还不清楚的学习者。

元框架最核心的差异不在“命令怎么运行”，而在页面、数据和运行环境的关系。理解这一点后，Nuxt 和 Next 的很多 API 都会变得更容易判断。

## 一句话理解

普通 SPA 常见流程：

```text
浏览器加载空 HTML
↓
下载 JS
↓
前端路由决定页面
↓
浏览器请求数据
↓
渲染内容
```

元框架可以变成：

```text
服务器知道当前路由
↓
服务器提前获取数据
↓
服务器生成 HTML
↓
浏览器接管交互
```

这就是为什么元框架更适合 SEO、内容站、首屏性能和复杂部署策略。

## 文件路由

文件路由的核心是：目录结构就是 URL 结构。

Nuxt 示例：

```text
pages/index.vue          -> /
pages/about.vue          -> /about
pages/products/[id].vue  -> /products/:id
```

Next 示例：

```text
app/page.tsx             -> /
app/about/page.tsx       -> /about
app/products/[id]/page.tsx -> /products/:id
```

优点：

- 路由结构容易看懂。
- 页面和路由绑定紧密。
- 大项目里不需要维护巨大的路由表。

注意：

- 文件命名会直接影响 URL。
- 重命名文件就是改路由。
- 动态参数要和数据获取逻辑一起检查。

## 布局

布局用于承载多个页面共享的结构。

适合放：

- 顶部导航。
- 侧边栏。
- 页脚。
- 登录后应用壳。
- 内容宽度和基础容器。

不适合放：

- 某个页面独有的请求。
- 某个业务表单的状态。
- 临时弹窗逻辑。

布局的职责是稳定的页面框架，不是业务逻辑垃圾桶。

## 数据获取位置

先按边界决定数据放在哪里：路由解析参数，服务端完成鉴权与首屏数据，页面输出可流式内容，客户端组件只接管局部交互。

把所有请求都搬到客户端会失去首屏和服务端权限优势；把播放器、拖拽等高频交互都放在服务端也会制造不必要往返。

元框架里最重要的判断是：数据在哪里取？

| 位置 | 适合场景 | 注意 |
| --- | --- | --- |
| 服务端获取 | SEO、首屏、私密 token、数据库访问 | 注意缓存和用户态 |
| 客户端获取 | 强交互、登录后动态数据、浏览器 API | 首屏可能先空白 |
| 构建时获取 | 内容站、文档、博客 | 数据更新需要重新生成或增量更新 |
| API route 获取 | BFF、聚合接口、隐藏私密配置 | 需要鉴权和日志 |

不要所有数据都放客户端取，也不要所有数据都放服务端取。判断依据是页面目标和数据性质。

## 加载和错误状态

元框架通常提供专门的加载和错误机制。

你需要为这些状态设计页面：

```text
加载中
↓
成功
↓
空数据
↓
业务错误
↓
系统错误
↓
404
```

很多真实项目的问题不是“接口失败”，而是失败时没有清晰 UI，用户只看到白屏。

## 用户态和权限

登录后页面最容易出问题，因为服务端和客户端都可能参与渲染。

需要确认：

- token 存在 cookie 还是 localStorage。
- 服务端是否能读取登录态。
- 页面是否可以被公共缓存。
- 权限不足时返回 403 还是重定向。
- 刷新页面后权限是否恢复。

如果登录态只存在 localStorage，服务端渲染时通常读不到它。此时不要假设服务端能知道当前用户。

## 缓存判断

每个页面都问：

```text
这个页面是否所有用户看到一样？
数据多久变化一次？
是否允许用户看到旧数据？
用户操作后是否必须立刻更新？
```

示例：

| 页面 | 缓存倾向 |
| --- | --- |
| 官网首页 | 可缓存 |
| 博客详情 | 可缓存或静态生成 |
| 商品详情 | 可缓存但要考虑库存和价格 |
| 个人中心 | 不应公共缓存 |
| 后台审批列表 | 动态请求 |

## 实际项目常见问题

### 问题 1：服务端渲染和客户端显示不一致

常见原因：

- 服务端生成随机值，客户端又生成一次。
- 服务端和客户端时区不同。
- 首屏依赖浏览器 API。
- 登录态在服务端不可见。

处理：

- 随机值和时间要固定或只在客户端生成。
- 浏览器 API 放到客户端生命周期里。
- 用户态页面谨慎使用公共缓存。

### 问题 2：页面缓存导致数据不更新

常见原因：

- 页面被静态生成。
- fetch 或框架缓存策略没有配置清楚。
- CDN 缓存了响应。

处理：

- 明确页面是否需要动态渲染。
- 设置合适 revalidate 或缓存头。
- 用户操作后做重新验证或刷新数据。

### 问题 3：API 密钥泄漏到浏览器

常见原因：

- 把私密变量写进公开环境变量。
- 在客户端组件中调用只能服务端使用的接口。

处理：

- 密钥只放服务端 runtime config 或环境变量。
- 客户端通过 server api 或 route handler 间接访问。
