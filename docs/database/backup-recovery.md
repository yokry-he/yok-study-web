# 备份与恢复

## 适合谁看

适合已经会写表结构和 CRUD，但还没有系统处理过数据备份、恢复和上线前数据风险的人：

- 只知道“数据库有备份”，但不知道能不能恢复。
- 上线前不备份，出问题后才找运维。
- 误删数据后不知道应该停写、导出还是回滚。
- 备份文件没有加密、没有过期策略。
- 从来没有演练过恢复。

备份不是“存一份文件”这么简单。真正有价值的是：知道备份覆盖什么、多久备一次、保存在哪里、谁能访问、多久能恢复、恢复后如何验证。

## 先理解两个目标

| 目标 | 含义 | 例子 |
| --- | --- | --- |
| RPO | 最多能接受丢失多久的数据 | 5 分钟、1 小时、1 天 |
| RTO | 最多能接受多久恢复服务 | 10 分钟、1 小时、半天 |

如果业务是后台配置系统，RPO 可能可以是 1 小时。如果是订单或支付系统，RPO 可能要接近 0。

不要只说“要备份”，要说清楚能接受丢多少、多久恢复。

## 备份类型

| 类型 | 说明 | 适合 |
| --- | --- | --- |
| 逻辑备份 | 导出 SQL 或可恢复的逻辑数据 | 中小库、迁移、单表恢复 |
| 物理备份 | 复制数据库底层文件或快照 | 大库、快速恢复 |
| 全量备份 | 备整个库 | 周期性基线 |
| 增量备份 | 只备变化部分 | 大数据量、降低成本 |
| 时间点恢复 | 恢复到某个具体时间点 | 误删、错误迁移 |

初学项目可以先掌握逻辑备份，再理解云数据库快照和时间点恢复。

## PostgreSQL 逻辑备份

PostgreSQL 常用 `pg_dump`。

示例：

```bash
pg_dump -h 127.0.0.1 -U app_user -d app_db -Fc -f app_db.dump
```

常见参数：

| 参数 | 说明 |
| --- | --- |
| `-h` | 数据库地址 |
| `-U` | 用户名 |
| `-d` | 数据库名 |
| `-Fc` | custom 格式，便于配合 pg_restore |
| `-f` | 输出文件 |

恢复示例：

```bash
pg_restore -h 127.0.0.1 -U app_user -d app_db --clean --if-exists app_db.dump
```

恢复前一定确认目标库。很多事故来自把备份恢复到了错误环境。

## MySQL 逻辑备份

MySQL 常用 `mysqldump`。

示例：

```bash
mysqldump -h 127.0.0.1 -u app_user -p --single-transaction app_db > app_db.sql
```

常见参数：

| 参数 | 说明 |
| --- | --- |
| `--single-transaction` | 对 InnoDB 表做一致性备份，减少锁表影响 |
| `--routines` | 导出存储过程和函数 |
| `--triggers` | 导出触发器 |
| `--events` | 导出事件 |

恢复示例：

```bash
mysql -h 127.0.0.1 -u app_user -p app_db < app_db.sql
```

恢复前同样要确认目标库和当前数据状态。

## 备份前要确认什么

上线前或大迁移前，至少确认：

- 当前环境是生产、预发还是测试。
- 备份命令连接的是正确数据库。
- 备份用户有足够权限。
- 备份文件保存路径可用。
- 磁盘空间足够。
- 备份过程不会长时间锁表。
- 备份文件是否包含敏感数据。

不要在不清楚连接串的情况下直接执行备份或恢复命令。

## 恢复比备份更重要

没有演练过恢复的备份，只能算“可能有用”。

恢复演练应该包含：

1. 选择一份真实备份。
2. 恢复到临时库。
3. 执行结构检查。
4. 执行业务抽样检查。
5. 对比关键表数量。
6. 验证应用能连接。
7. 记录恢复耗时。

验证 SQL 示例：

```sql
select count(*) from users;
select count(*) from orders;
select max(created_at) from audit_log;
```

恢复后要确认数据不是“能启动就行”，而是关键业务数据真的存在。

## 误删数据怎么办

发现误删后不要立刻乱操作。

推荐流程：

```text
停止相关写入
↓
确认影响范围
↓
保留当前现场
↓
找到最近可用备份
↓
恢复到临时库
↓
抽取误删数据
↓
评审修复 SQL
↓
回写生产
↓
补审计和复盘
```

如果直接把整个备份覆盖生产，可能会丢掉误删之后的新数据。很多时候更安全的方式是恢复到临时库，再把需要的数据补回生产。

## 备份文件也需要安全治理

备份文件通常包含完整业务数据，比线上库更容易被忽视。

必须考虑：

- 存储加密。
- 传输加密。
- 访问权限。
- 下载审计。
- 过期删除。
- 脱敏备份。
- 不把备份文件提交到 Git。

开发环境使用生产备份时，要先脱敏。手机号、邮箱、身份证、地址、Token、密码哈希都属于敏感数据。

## 备份计划怎么定

一个简单可执行的计划：

| 类型 | 频率 | 保存时间 | 用途 |
| --- | --- | --- | --- |
| 每日全量 | 每天凌晨 | 7 到 30 天 | 常规恢复 |
| 迁移前备份 | 每次结构变更前 | 至少覆盖回滚窗口 | 上线失败恢复 |
| 关键表导出 | 重要操作前 | 按业务决定 | 局部修复 |
| 云数据库快照 | 按业务等级 | 按成本和合规决定 | 快速整库恢复 |

核心系统还需要 binlog、WAL 或云厂商提供的时间点恢复能力。

## 实际项目问题

### 1. 备份文件存在，但恢复失败

**原因**

- 备份文件不完整。
- 目标数据库版本不兼容。
- 缺少扩展、角色或权限。
- 文件损坏。
- 恢复命令没演练过。

**解决方案**

- 定期恢复到临时库。
- 记录数据库版本和扩展。
- 保存恢复命令。
- 备份完成后校验文件大小和校验和。

### 2. 迁移失败后不知道怎么回滚

**原因**

上线前只写了 migration，没有备份和验证 SQL。

**解决方案**

- 大迁移前先备份。
- 迁移说明写清楚回滚风险。
- destructive migration 分阶段执行。
- 删除字段前先停止读取，再观察，再删除。

### 3. 测试环境使用生产数据泄漏隐私

**原因**

备份直接导入测试环境，没有脱敏。

**解决方案**

- 建立脱敏脚本。
- 替换手机号、邮箱、姓名、地址等字段。
- 清理 Token、密钥、第三方账号。
- 限制测试环境数据库访问。

### 4. 只备份数据库，忽略文件和对象存储

**原因**

业务数据分散在数据库和对象存储里。

**解决方案**

- 数据库记录和文件对象要一起纳入备份计划。
- 记录对象 key、文件状态和业务表之间的对应关系。
- 恢复演练必须包含文件访问验证。

## 最佳实践

- 先定义 RPO 和 RTO，再设计备份策略。
- 每次高风险迁移前必须备份。
- 定期做恢复演练。
- 恢复优先恢复到临时库，再评估是否回写生产。
- 备份文件必须加密、限权、审计和过期清理。
- 生产备份进入非生产环境前必须脱敏。
- 备份策略要写进项目文档和上线检查清单。
