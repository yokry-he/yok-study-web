# 数据库常见问题

## 使用方式

数据库问题排查要先分层：

```text
应用代码
↓
SQL 写法
↓
索引和执行计划
↓
事务和锁
↓
连接池
↓
数据库资源
↓
缓存一致性
```

不要一看到慢就加索引，也不要一看到错误就重启数据库。

## 1. 接口突然变慢

**排查**

1. 是所有接口慢，还是某个接口慢。
2. 慢接口是否访问数据库。
3. SQL 是否变化。
4. 数据量是否增长。
5. 执行计划是否变了。
6. 数据库 CPU、IO、连接数是否异常。

**解决方案**

对具体 SQL 使用 EXPLAIN：

```sql
explain (analyze, buffers)
select *
from orders
where customer_id = 1001
order by created_at desc
limit 20;
```

先找到瓶颈，再决定是改 SQL、加索引、分页、缓存还是拆表。

## 2. 数据量一大列表就慢

**常见原因**

- 没有索引。
- offset 太大。
- 返回字段太多。
- 关联查询产生 N+1。
- 排序字段没有索引支持。

**解决方案**

- 为筛选和排序设计复合索引。
- 大分页改游标分页。
- 列表只返回必要字段。
- 批量查询关联数据。

## 3. 连接数耗尽

**现象**

应用报连接超时、too many connections、remaining connection slots are reserved 等错误。

**常见原因**

- 每次请求都创建新连接，没有复用连接池。
- 连接池设置过大，多个实例叠加超过数据库上限。
- 慢查询占住连接。
- 事务未提交导致连接长期占用。

**解决方案**

- 应用使用连接池。
- 根据实例数计算总连接数。
- 设置连接超时和空闲释放。
- 排查长事务和慢查询。
- PostgreSQL 高并发场景考虑连接池代理。

## 4. 死锁

**现象**

数据库报 deadlock detected 或 lock wait timeout。

**原因**

多个事务以不同顺序锁定资源。

**解决方案**

- 统一资源访问顺序。
- 缩短事务。
- 批量更新按主键排序。
- 捕获死锁并安全重试。
- 记录事务内执行的 SQL。

## 5. 唯一约束冲突

**现象**

创建用户时报用户名或手机号重复。

**正确理解**

唯一约束冲突不是数据库异常，而是业务规则被触发。

**解决方案**

前端可以提前校验，但最终以数据库唯一约束为准。后端捕获冲突并返回明确业务错误：

```text
用户名已存在
手机号已绑定
```

## 6. 迁移在测试环境成功，生产失败

**常见原因**

- 生产有历史脏数据。
- 生产表更大，添加索引超时。
- 生产约束和测试不一致。
- 迁移依赖的扩展或权限不存在。

**解决方案**

上线前准备检查 SQL：

```sql
select email, count(*)
from users
group by email
having count(*) > 1;
```

如果要加唯一约束，必须先确认没有重复数据。

## 7. 缓存和数据库不一致

**常见原因**

- 更新数据库后忘记删缓存。
- 删除缓存失败但没有补偿。
- 多个服务使用不同 key 规则。
- 缓存过期时间太长。

**解决方案**

- 统一 key 生成。
- 写库成功后删除缓存。
- 关键缓存加短 TTL。
- 对高风险场景加消息补偿或定期校验。

## 8. Redis 热点 key 压力大

**现象**

某个热门商品、活动或排行榜 key 请求量极高。

**解决方案**

- 本地短缓存。
- 热点 key 拆分。
- 请求合并。
- 限流和降级。
- 提前预热。

## 9. 删除数据后发现报表不对

**原因**

物理删除了仍有分析价值或审计价值的数据。

**解决方案**

核心业务数据使用软删除，并在报表 SQL 中明确是否包含删除数据。

```sql
deleted_at timestamptz null
```

为空表示正常，有值表示删除时间。

## 10. ORM 生成的 SQL 很慢

**原因**

ORM 隐藏了实际 SQL，开发者只看代码调用，不看数据库执行。

**解决方案**

- 打开 SQL 日志。
- 复制慢 SQL 到数据库执行 EXPLAIN。
- 避免循环里触发关联查询。
- 对复杂报表写明确 SQL。

## 排查清单

| 问题 | 优先看 |
| --- | --- |
| 慢查询 | EXPLAIN、索引、扫描行数 |
| 死锁 | 事务顺序、锁等待、重试策略 |
| 连接耗尽 | 连接池、慢查询、长事务 |
| 迁移失败 | 历史数据、锁、权限、扩展 |
| 缓存不一致 | key 规则、删除缓存、TTL |
| 数据错乱 | 约束、事务、幂等、审计日志 |

## 最佳实践

- 慢查询必须保留 SQL 和执行计划。
- 数据库错误要转成明确业务错误。
- 所有核心表保留创建时间和更新时间。
- 权限、订单、财务等核心数据保留审计记录。
- 迁移失败不要手工乱改生产库，先备份和评估影响。
- 数据库结构和变更原因必须写文档。
