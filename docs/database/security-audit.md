# 数据安全、审计与脱敏

## 适合谁看

适合开始接触用户数据、权限数据、订单数据、财务数据或企业后台系统的人：

- 数据库账号所有人共用。
- 生产库可以被开发随意连接。
- 审计日志只记了“操作成功”，没有上下文。
- 测试环境直接使用生产数据。
- 删除、导出、改权限等高风险操作无法追溯。

数据库安全不是只靠“设置一个复杂密码”。它包含账号权限、网络边界、敏感数据、审计日志、备份治理、变更审批和异常排查。

## 安全边界

数据库安全至少分为五层：

```text
网络访问
↓
账号权限
↓
数据读写
↓
审计记录
↓
备份和导出
```

任何一层失守，都可能导致数据泄漏或无法追责。

## 最小权限原则

不同用途使用不同账号：

| 账号 | 权限 | 用途 |
| --- | --- | --- |
| app_user | 只允许应用需要的读写权限 | 后端服务连接 |
| migration_user | 允许结构变更 | migration 执行 |
| readonly_user | 只读 | 报表、排查、临时查询 |
| backup_user | 备份所需权限 | 备份任务 |
| admin_user | 管理权限 | 少数管理员使用 |

不要让应用使用数据库超级管理员账号。

原因：

- 应用漏洞会放大成数据库管理权限。
- 误操作更难限制。
- 审计时无法区分行为来源。
- 密码泄漏后影响范围过大。

## 生产库访问

生产库访问应该有明确流程：

1. 提交访问原因。
2. 说明要查哪些表。
3. 限定时间窗口。
4. 使用只读账号优先。
5. 查询语句留痕。
6. 高风险操作需要审批。

临时排查不要直接给长期账号。可以使用有过期时间的临时凭证，或通过堡垒机、云厂商审计通道连接。

## 敏感字段识别

常见敏感数据：

| 类型 | 示例 |
| --- | --- |
| 身份信息 | 姓名、身份证、护照 |
| 联系方式 | 手机号、邮箱、地址 |
| 账号安全 | 密码哈希、Token、密钥 |
| 财务数据 | 金额、银行卡、发票 |
| 企业数据 | 合同、组织结构、员工信息 |
| 行为数据 | 登录日志、访问记录、定位 |

敏感字段应该在数据字典或表注释里标记清楚。

示例：

```sql
comment on column user_profile.phone is
  '手机号，个人敏感信息；用于登录验证和通知，不允许在非授权接口明文返回';
```

字段注释不是形式主义。它能帮助后续开发知道哪些字段不能随便展示、导出或打印日志。

## 明文、哈希和加密

不同数据处理方式不同：

| 方式 | 适合 | 说明 |
| --- | --- | --- |
| 明文 | 非敏感展示字段 | 例如文章标题、公开名称 |
| 哈希 | 密码、不可逆校验 | 密码只能哈希，不能加密后可解密 |
| 加密 | 需要恢复原文的敏感数据 | 例如某些证件号、业务密钥 |
| 脱敏 | 展示或测试环境 | 例如 `138****0000` |

密码不要自己设计算法。应使用成熟密码哈希方案，并配置合理成本。

## 接口返回要脱敏

数据库里保存了手机号，不代表接口可以原样返回。

推荐在 service 或 serializer 层处理：

```ts
function maskPhone(phone: string) {
  return phone.replace(/(\\d{3})\\d{4}(\\d{4})/, '$1****$2')
}

function toUserProfileView(user: UserRecord) {
  return {
    id: user.id,
    displayName: user.displayName,
    phone: maskPhone(user.phone)
  }
}
```

不要把数据库记录直接 `return` 给前端。

## 日志不要泄漏敏感数据

不要记录：

- 密码。
- Token。
- Cookie。
- 身份证。
- 完整手机号。
- 完整银行卡号。
- 大量用户原始数据。

错误日志应该帮助定位问题，但不应该变成数据泄漏源。

推荐日志包含：

- request id。
- 操作人 id。
- 操作类型。
- 目标资源 id。
- 错误码。
- 耗时。

敏感字段要脱敏或不记录。

## 审计日志记录什么

审计日志不是普通业务日志。它用于追踪高风险行为。

建议记录：

| 字段 | 说明 |
| --- | --- |
| id | 审计记录主键 |
| action | 操作类型，例如 `user.delete` |
| operator_id | 操作人 |
| target_type | 操作对象类型 |
| target_id | 操作对象 ID |
| before_snapshot | 修改前关键字段 |
| after_snapshot | 修改后关键字段 |
| ip | 操作来源 |
| user_agent | 客户端信息 |
| request_id | 关联请求 |
| created_at | 操作时间 |

示例：

```sql
create table audit_log (
  id bigserial primary key,
  action varchar(120) not null,
  operator_id bigint null,
  target_type varchar(80) not null,
  target_id varchar(120) not null,
  before_snapshot jsonb null,
  after_snapshot jsonb null,
  ip varchar(64) null,
  user_agent text null,
  request_id varchar(120) null,
  created_at timestamptz not null default now()
);

comment on table audit_log is
  '审计日志，记录高风险业务操作，用于追溯权限、数据变更和安全事件';

comment on column audit_log.action is
  '操作类型，使用稳定业务编码，例如 user.delete、role.permission.update';
```

审计日志要避免被普通业务流程随意删除。

## 哪些操作必须审计

建议审计：

- 登录失败次数异常。
- 修改用户状态。
- 修改角色权限。
- 导出数据。
- 删除数据。
- 修改价格、库存、金额。
- 变更组织和员工关系。
- 执行高风险迁移或脚本。
- 管理员查看敏感详情。

不是所有读操作都要写审计，否则成本很高。重点是高风险、高价值、高合规要求的操作。

## 数据脱敏

脱敏用于展示、导出、测试环境和日志。

常见策略：

| 数据 | 脱敏方式 |
| --- | --- |
| 手机号 | 保留前三后四 |
| 邮箱 | 保留用户名首尾和域名 |
| 身份证 | 只保留前六后四 |
| 姓名 | 保留姓氏 |
| 地址 | 保留城市，隐藏详细地址 |
| Token | 不展示或只展示后四位 |

测试环境不要直接导入生产明文数据。

## 备份和导出同样要审计

很多泄漏不是从在线数据库发生，而是从导出的 SQL、Excel、CSV、日志包发生。

必须记录：

- 谁导出了数据。
- 导出了哪些范围。
- 导出原因。
- 文件保存在哪里。
- 文件多久过期。
- 是否脱敏。

导出文件要有过期机制，不要永久留在服务器或对象存储里。

## 实际项目问题

### 1. 开发直接连生产库查问题

**风险**

- 查询误操作。
- 数据泄漏。
- 无法追踪谁查了什么。

**解决方案**

- 默认只给只读账号。
- 通过审批和审计通道访问。
- 高风险查询在预发或临时库复现。
- 排查完成后回收临时权限。

### 2. 前端隐藏按钮，但接口仍能调用

**风险**

用户可以绕过前端直接请求接口。

**解决方案**

- 权限判断必须在后端。
- 数据库层保存稳定权限编码。
- 高风险操作写审计日志。
- 无权限返回 403。

### 3. 日志里出现 Token

**风险**

日志平台、报错平台、截图都可能泄漏 Token。

**解决方案**

- 请求日志过滤 Authorization、Cookie。
- 错误对象序列化时过滤敏感字段。
- 日志平台设置访问权限。
- 泄漏后立即吊销相关 Token。

### 4. 审计日志太多，没人看

**原因**

什么都记录，但没有风险分级和查询入口。

**解决方案**

- 只把关键行为作为审计日志。
- 普通访问日志和审计日志分开。
- 给管理员提供按操作人、资源、时间、动作查询的页面。
- 对异常行为做告警，例如短时间大量导出。

## 最佳实践

- 应用账号不能使用数据库超级管理员权限。
- 生产访问要审批、限时、留痕。
- 敏感字段在表注释或数据字典中明确标记。
- 接口返回、日志、导出都要做脱敏。
- 权限判断必须在后端执行。
- 高风险操作要有审计日志。
- 备份文件和导出文件必须加密、限权和过期。
- 测试环境使用生产数据前必须脱敏。
