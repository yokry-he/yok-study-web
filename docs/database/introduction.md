# 数据库学习导览

## 适合谁看

适合已经能写前端页面、Node.js API 或后台管理系统，但对数据库设计、索引、事务、迁移和缓存还没有系统认识的人。

数据库不是“会写增删改查”就够了。实际项目中，很多线上问题都来自数据库层：

- 表结构一开始没设计好，后面改动困难。
- 没有索引，数据量上来后接口变慢。
- 事务范围太大，造成锁等待和死锁。
- 缓存没有过期策略，用户看到旧数据。
- 迁移没有备注，半年后没人敢改表。
- 删除数据没有软删除或审计，问题无法追溯。

## 学习目标

学完数据库模块第一版，你应该能做到：

- 知道 MySQL、PostgreSQL、Redis 的典型使用场景。
- 能设计一张业务表，并写清楚字段、约束、索引和业务含义。
- 能根据查询条件判断是否需要索引。
- 能理解事务、锁、隔离级别和并发风险。
- 能写可回溯、可审查的数据库迁移说明。
- 能排查慢查询、N+1 查询、连接数耗尽和缓存不一致。

## 数据库分类

| 类型 | 常见产品 | 适合场景 |
| --- | --- | --- |
| 关系型数据库 | MySQL、PostgreSQL | 用户、订单、权限、财务、强一致业务 |
| 键值缓存 | Redis | 缓存、会话、限流、排行榜、分布式锁 |
| 文档数据库 | MongoDB | 半结构化文档、内容配置、灵活字段 |
| 搜索引擎 | Elasticsearch、OpenSearch | 全文搜索、日志检索 |
| 时序数据库 | TimescaleDB、InfluxDB | 指标、监控、时间序列 |

初学者优先掌握关系型数据库和 Redis。它们覆盖了多数 Web 项目的核心需求。

## SQL 和 NoSQL 怎么选

优先选择关系型数据库的情况：

- 数据之间有明确关系。
- 需要事务。
- 需要唯一约束、外键、检查约束。
- 需要稳定查询和报表。
- 需要强一致性。

可以考虑 Redis 或其他 NoSQL 的情况：

- 数据只需要短期保存。
- 读写频率非常高。
- 可以接受缓存失效和重建。
- 数据结构更像 key-value、集合、队列或排行榜。

不要因为“关系型数据库麻烦”就把核心业务数据放进缓存。缓存可以丢，数据库不应该随便丢。

## 最重要的工程原则

### 表结构必须有文档

每个表至少要写清楚：

- 表解决什么业务问题。
- 每个字段的业务含义。
- 字段是否允许为空。
- 默认值为什么这样设置。
- 唯一约束和外键约束的原因。
- 索引对应哪些查询场景。
- 迁移为什么发生。

数据库结构一旦上线，就会影响数据、接口、权限、报表和运维。没有说明的字段会成为后续维护成本。

### 不要过早优化，但要保留优化证据

索引不是越多越好。索引会提升查询，但也会增加写入和存储成本。

正确做法：

1. 先明确查询场景。
2. 根据 WHERE、JOIN、ORDER BY 设计索引。
3. 用 EXPLAIN 或慢查询日志验证。
4. 在迁移说明里写清楚索引原因。

### 数据库约束不是可选项

只靠后端代码校验不够。后端可能有 bug，脚本可能绕过接口，后续服务可能复用数据库。

数据库层应该兜底：

- 主键保证记录可定位。
- 唯一约束保证业务唯一。
- 非空约束保证关键字段存在。
- 外键或业务约束保证关系可信。
- 检查约束保证值域合理。
