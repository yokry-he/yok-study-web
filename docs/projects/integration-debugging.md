# 前后端联调排查

## 适合谁看

适合前端、后端或全栈开发在接口联调时遇到这些问题：

- 前端说“后端接口不对”。
- 后端说“前端参数没传”。
- 本地 mock 正常，真实接口失败。
- 登录后仍然 401。
- 列表字段对不上。
- 上传、导出、分页、权限联调反复返工。

联调排查的关键不是互相猜，而是用请求证据说话。

## 联调最小证据

每个接口问题至少保留：

| 证据 | 说明 |
| --- | --- |
| 请求 URL | 完整路径和 query |
| 请求方法 | GET、POST、PUT、DELETE |
| 请求头 | Authorization、Cookie、Content-Type |
| 请求体 | JSON、FormData 或文件 |
| 响应状态码 | 200、400、401、403、500 |
| 响应体 | 错误码、message、data |
| request id | 关联后端日志 |

没有这些证据，就很容易变成口头争论。

## 排查顺序

```text
浏览器 Network
↓
请求 URL 和方法
↓
请求头和认证信息
↓
请求参数位置
↓
响应状态码和错误体
↓
后端日志 request id
↓
数据库或第三方服务
```

不要直接跳到“是不是后端 bug”。先确认请求是否真的和接口约定一致。

## 常见问题 1：参数位置错了

现象：

- 前端传了参数。
- 后端说没收到。

常见原因：

| 前端传法 | 后端读取 |
| --- | --- |
| query | `req.query` |
| path params | `req.params` |
| JSON body | `req.body` |
| form-data | multipart parser |
| header | `req.headers` |

解决：

- 接口文档写清参数位置。
- 前端请求封装不要自动改变结构。
- 后端参数校验返回明确字段错误。

## 常见问题 2：Content-Type 不一致

现象：

- JSON 接口后端收不到 body。
- 文件上传接口 400。

原因：

- JSON 接口没有 `Content-Type: application/json`。
- 文件上传手动写了错误 boundary。
- 后端没有配置 body parser 或 multipart parser。

解决：

- JSON 用统一 request 封装。
- 上传文件时让浏览器自动设置 multipart boundary。
- 后端按接口类型配置解析中间件。

## 常见问题 3：登录后仍然 401

现象：

- 登录接口成功。
- 调其他接口仍然未登录。

原因：

- Token 没保存。
- 请求没带 Authorization。
- Cookie 没带上。
- 跨域请求缺少 credentials。
- 服务端 session 过期。

解决：

- Network 里确认请求头或 Cookie。
- 前端统一注入认证信息。
- Cookie 登录态检查 SameSite、Domain、Path、Secure。
- 后端返回 401 时前端统一跳登录。

## 常见问题 4：无权限应该是 403

现象：

- 用户已登录，但接口返回 401 或 500。

原因：

- 后端把未登录和无权限混在一起。
- 权限码不存在时默认放行或默认报错。
- 前端只隐藏按钮，没有处理接口 403。

解决：

- 未登录返回 401。
- 无权限返回 403。
- 前端统一展示无权限提示。
- 后端记录权限判断上下文。

## 常见问题 5：分页字段不一致

现象：

- 前端分页器页码异常。
- 删除最后一条后停在空页。
- 总数显示不对。

统一约定：

```json
{
  "list": [],
  "page": 1,
  "pageSize": 20,
  "total": 128
}
```

建议：

- 前端统一适配分页结构。
- 后端返回当前页和总数。
- 删除后如果当前页无数据，前端回退上一页重新查。

## 常见问题 6：枚举值对不上

现象：

- 后端返回 `enabled`，前端按 `1` 判断。
- 前端展示未知状态。

解决：

- 枚举集中维护。
- 接口文档写清值域。
- TypeScript 类型和后端 DTO 同步。
- 未知枚举展示兜底文案。

## 常见问题 7：时间格式不一致

现象：

- 列表时间差 8 小时。
- 前端排序不正确。

解决：

- 后端统一返回 ISO 字符串或约定格式。
- 前端只在展示层格式化。
- 不要把展示格式再提交回后端。
- 涉及时区时文档写清楚。

## 常见问题 8：导出接口失败

现象：

- 浏览器下载的是乱码。
- 文件名不对。
- 错误时仍然当文件下载。

解决：

- 前端按 blob 处理成功响应。
- 错误响应和文件响应区分。
- 后端设置 `Content-Disposition`。
- 导出任务耗时长时改成异步任务。

## 联调记录模板

```md
## 接口名称

请求方法：
请求路径：
请求参数：
认证方式：
成功响应：
失败响应：
联调问题：
最终约定：
```

联调完成后，把成功样例和失败样例都保存下来。失败样例比成功样例更能减少返工。

## 最佳实践

- 联调问题先看 Network，不靠猜。
- 每个请求带 request id。
- 参数位置、字段名、枚举、分页结构提前约定。
- 401、403、400、409、500 要有清晰边界。
- 文件上传和导出单独写联调样例。
- 联调发现的问题要回写接口文档和检查清单。
