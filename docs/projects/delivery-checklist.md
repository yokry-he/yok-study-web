# 项目交付检查清单

## 适合谁看

这篇适合准备把项目从“能跑”推进到“能交付”的同学。它尤其适合下面场景：

- Vue Admin 或中后台项目准备上线。
- 前后端联调经常返工。
- 上线后才发现环境变量、路由、缓存或权限问题。
- 团队没有固定的验收、回滚和故障复盘流程。

学习文档解决“怎么写”，检查清单解决“交付前怎么确认没有漏”。真实项目里，很多事故不是不会写代码，而是没有在正确时间检查正确问题。

## 使用方式

每次交付至少走四个阶段：

```text
开发完成
↓
联调验收
↓
上线发布
↓
上线后观察
↓
问题复盘
```

不要等所有事情都结束才补文档。每次发现一个遗漏，都把它加回检查清单或 [真实项目问题库](/projects/real-world-issues)。

## 阶段一：开发完成检查

### 页面和交互

| 检查项 | 通过标准 | 不通过时看哪里 |
| --- | --- | --- |
| 页面能从菜单进入 | 菜单、面包屑、当前激活状态正确 | [Vue 路由与页面](/vue/router) |
| 页面刷新不丢状态 | 直接刷新后能正常恢复登录态、菜单和权限 | [前端页面与状态问题](/projects/issues-frontend) |
| 表单可提交、可取消、可回填 | 新增、编辑、详情三种状态互不污染 | [Vue 表单处理](/vue/forms) |
| 列表支持空状态 | 无数据、加载中、错误状态都有明确展示 | [Vue 请求与接口封装](/vue/request) |
| 删除、批量删除处理分页 | 删除最后一条后不会停在空页 | [真实项目问题库](/projects/real-world-issues) |

### 状态和权限

权限相关页面至少检查：

- 没有权限时菜单不显示或进入 403。
- 没有按钮权限时按钮不显示或禁用。
- 刷新页面后权限仍然正确。
- 权限码不存在时不应默认放行。
- 管理员、普通用户、只读用户分别验证。

权限码不要散落在页面里，推荐集中维护。

```ts
export const PermissionCode = {
  UserCreate: 'system:user:create',
  UserUpdate: 'system:user:update',
  UserDelete: 'system:user:delete'
} as const
```

### 样式和响应式

至少检查两个宽度：

```text
桌面端：1440px
窄屏：390px
```

重点看：

- 是否存在横向滚动。
- 表格操作列是否被压扁。
- 图标、头像、状态点是否变形。
- 弹窗和抽屉是否超出屏幕。
- 组件库按钮、开关、输入框尺寸是否异常。

如果发现组件样式异常，先查业务 CSS 是否使用了宽泛选择器，不要直接叠更高优先级覆盖。

## 阶段二：前后端联调检查

### 接口契约

每个接口至少确认：

| 项目 | 要确认什么 |
| --- | --- |
| method | GET、POST、PUT、DELETE 是否一致 |
| path | 路径是否包含正确版本和业务前缀 |
| query | 查询参数名称、类型、默认值 |
| body | JSON、表单、文件上传格式 |
| response | 成功结构、分页结构、空数据结构 |
| error | 错误码、错误信息、字段级错误 |
| auth | token、cookie、权限码、租户信息 |

联调中最常见的问题是前端和后端都“看起来没错”，但参数位置、字段名或错误码约定不一致。遇到这类问题，先看 [后端接口与服务问题](/projects/issues-backend)。

### 请求和错误处理

前端请求封装至少要覆盖：

- 统一 baseURL。
- token 或 cookie 处理。
- 401 只处理一次，避免重复弹窗。
- 403 有明确提示或跳转。
- 业务错误不当成系统异常。
- 下载、上传、分页、列表查询各自有清晰封装。

错误响应建议使用稳定结构。

```json
{
  "code": "USER_MOBILE_DUPLICATED",
  "message": "手机号已被其他用户使用",
  "requestId": "req_123"
}
```

前端不要用中文文案判断业务分支，要用稳定错误码。

### 数据一致性

涉及多步写入时确认：

- 是否需要事务。
- 是否需要幂等。
- 是否可能重复提交。
- 是否需要唯一索引。
- 是否会更新缓存。
- 是否有异步任务或消息队列。

典型高风险场景：

- 创建订单。
- 扣减库存。
- 分配角色权限。
- 批量导入。
- 审批流转。
- 支付和回调。

这些问题可以对照 [数据库与缓存问题](/projects/issues-database)。

## 阶段三：上线发布检查

### 构建检查

上线前必须在干净环境执行：

```bash
npm run docs:check
npm run docs:build
```

业务项目还应补充：

```bash
npm run lint
npm run test
npm run build
```

如果构建成功但页面白屏，优先检查：

- 静态资源是否 404。
- `base` 是否和部署路径一致。
- 环境变量是否正确。
- `index.html` 是否被强缓存。
- Console 第一条错误是什么。

### 路由和部署路径

使用 history 路由的前端项目，上线后至少验证：

- 首页能打开。
- 菜单能跳转。
- 二级路由刷新不 404。
- 详情页链接复制到新标签页能打开。
- 子路径部署时资源路径正确。

Nginx 常见配置：

```nginx
location / {
  try_files $uri $uri/ /index.html;
}
```

子路径部署要同步配置前端 `base` 和 Nginx 回退路径。

### 缓存和版本

发布时重点确认：

| 文件 | 推荐缓存策略 |
| --- | --- |
| `index.html` | 不强缓存或短缓存 |
| 带 hash 的 JS/CSS | 长缓存 |
| 图片、字体 | 根据版本策略设置缓存 |
| runtime config | 不强缓存 |

如果用户上线后看到旧页面，优先看 [部署、缓存与 DevOps 问题](/projects/issues-deployment)。

### 回滚准备

上线前必须能回答：

- 当前版本号是什么。
- 上一个可回滚版本是什么。
- 回滚需要几分钟。
- 回滚会不会影响数据库结构。
- 回滚后缓存是否需要清理。
- 谁负责执行回滚，谁负责验证。

如果本次发布包含数据库迁移，必须额外确认：

- 迁移是否向后兼容。
- 是否有备份。
- 是否有回滚脚本。
- 老代码能否兼容新字段。
- 新代码能否兼容旧数据。

数据库迁移不要只写 SQL，还要写清楚业务含义、约束、影响范围和回滚策略。

## 阶段四：上线后观察

### 前 30 分钟

发布后前 30 分钟重点观察：

- 核心页面是否能打开。
- 登录、权限、列表、提交是否正常。
- 错误日志是否升高。
- 接口耗时是否升高。
- 资源 404 是否增多。
- 用户反馈是否集中在同一功能。

### 指标和日志

至少保留这些信息：

| 信息 | 作用 |
| --- | --- |
| requestId | 串起前端报错、后端日志和网关日志 |
| userId | 确认是否是特定用户或权限问题 |
| route | 确认问题页面 |
| api path | 确认问题接口 |
| status code | 区分 401、403、404、500 |
| latency | 判断是否是性能问题 |
| version | 判断是否和本次发布有关 |

前端错误上报中不要记录敏感数据，例如密码、token、身份证号、完整手机号。

### 快速止损

出现线上问题时，不要一上来就改代码。先判断应该止损还是修复：

| 情况 | 优先动作 |
| --- | --- |
| 大量用户无法登录 | 回滚或切换登录服务 |
| 某个非核心按钮报错 | 临时隐藏入口或修复后发布 |
| 静态资源 404 | 回滚入口文件或恢复旧资源 |
| 数据写错 | 停止写入入口，先保护数据 |
| AI 回答越权 | 立即关闭相关功能或收紧检索权限 |

## 阶段五：问题复盘

### 复盘不是追责

复盘要回答：

- 发生了什么。
- 谁受影响。
- 为什么没在上线前发现。
- 怎么修复。
- 如何避免再次发生。
- 哪个检查项要补充。

不要只写“开发不细心”。这类结论不能降低下次出错概率。

### 复盘模板

```text
标题：
发生时间：
发现方式：
影响范围：
用户现象：
直接原因：
深层原因：
修复过程：
回滚或止损：
最终修复：
预防动作：
需要补充的文档：
负责人：
完成时间：
```

### 沉淀到文档站

复盘完成后，把内容拆到三个地方：

| 内容 | 放哪里 |
| --- | --- |
| 具体故障案例 | [真实项目问题库](/projects/real-world-issues) |
| 某技术的通用原理 | 对应技术模块，例如 Vue、数据库、DevOps |
| 流程检查项 | 当前交付检查清单 |

这样问题不会只停留在聊天记录或会议纪要里，而是变成后续学习者能复用的知识。
