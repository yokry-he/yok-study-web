# 权限系统案例

## 适合谁看

适合正在做后台管理系统、SaaS 控制台、企业内部门户或任何需要菜单、按钮、接口权限的项目。

权限系统看起来只是“控制按钮显示”，但真实项目里它会影响：

- 登录态。
- 用户角色。
- 菜单生成。
- 动态路由。
- 按钮显示。
- 接口授权。
- 数据范围。
- 审计日志。

如果权限边界没设计好，后期每加一个页面都会返工。

## 业务目标

做一个后台权限系统，支持：

- 用户登录。
- 用户绑定角色。
- 角色绑定权限。
- 根据权限生成菜单。
- 根据权限控制按钮。
- 后端接口校验权限。
- 高风险操作写审计日志。

## 数据模型

```text
user
↓
user_role
↓
role
↓
role_permission
↓
permission
```

权限表建议区分：

| 字段 | 说明 |
| --- | --- |
| code | 稳定权限编码，例如 `user:create` |
| name | 展示名称，例如 `新增用户` |
| type | menu、button、api |
| parent_code | 权限树父节点 |
| route_path | 菜单权限对应路由 |
| enabled | 是否启用 |

权限 code 一旦上线要尽量稳定。展示名称可以改，code 改名需要兼容迁移。

## 前端权限边界

前端负责用户体验：

- 显示可访问菜单。
- 隐藏或禁用无权限按钮。
- 进入无权限页面时展示 403。
- 刷新后恢复菜单和权限状态。

前端不负责最终安全判断。

前端权限判断示例：

```ts
export function hasPermission(code: string) {
  const authStore = useAuthStore()
  return authStore.permissions.includes(code)
}
```

按钮组件：

```vue
<PermissionButton permission="user:create">
  新增用户
</PermissionButton>
```

## 后端权限边界

后端负责最终安全：

- 检查是否登录。
- 检查是否有权限。
- 检查是否能访问目标数据。
- 写审计日志。

接口示例：

```ts
app.delete(
  '/api/users/:id',
  authRequired,
  permissionRequired('user:delete'),
  deleteUserController
)
```

不要因为前端隐藏了按钮，就跳过后端授权。

## 菜单权限和按钮权限

菜单权限：

- 决定用户能看到哪些页面入口。
- 通常和路由绑定。
- 刷新后需要重新恢复。

按钮权限：

- 决定页面内能执行哪些动作。
- 例如新增、编辑、删除、导出。
- 不能只靠按钮隐藏，接口也要校验。

API 权限：

- 服务端真正执行授权。
- 可以和按钮权限同 code，也可以单独维护。
- 高风险接口必须记录审计。

## 数据范围权限

除了“能不能操作”，还要考虑“能操作哪些数据”。

常见数据范围：

- 只能看自己。
- 能看本部门。
- 能看本部门及子部门。
- 能看指定组织。
- 能看全部。

示例：

```text
用户有 user:read
但只能查看自己部门的用户
```

这时不能只检查 `user:read`，还要在 service 或 repository 中限制查询范围。

## 权限初始化流程

```text
用户登录
↓
获取用户信息
↓
获取权限码
↓
生成菜单
↓
注册动态路由
↓
进入页面
```

刷新页面时，要重新执行这套初始化。否则常见问题是：刷新后菜单丢失、按钮全隐藏或动态路由 404。

## 实际项目问题

### 1. 权限码散落在页面里

**现象**

每个页面都有很多字符串：

```vue
<button v-if=\"permissions.includes('user:create')\">新增</button>
```

**问题**

权限码改名时很难全量修改，也缺少统一解释。

**解决方案**

- 建立权限常量。
- 封装权限组件或指令。
- 权限码变更走兼容迁移。

### 2. 刷新后按钮权限丢失

**原因**

权限只保存在内存中，刷新后没有重新拉取。

**解决方案**

- 应用启动时恢复用户信息。
- 拉取权限后再生成菜单和路由。
- 恢复失败进入登录页或错误页。

### 3. 只有菜单权限，没有接口权限

**风险**

用户可以直接调用接口。

**解决方案**

- 后端接口必须做授权。
- 无权限返回 403。
- 删除、导出、授权等操作写审计日志。

### 4. 超级管理员绕过所有逻辑

**风险**

代码里到处写 `if isAdmin return true`，后续很难控制。

**解决方案**

- 超级管理员也通过统一权限策略处理。
- 特权行为写审计。
- 高风险操作可以二次确认。

## 权限测试清单

- 未登录访问页面。
- 已登录无菜单权限。
- 已登录有菜单但无按钮权限。
- 直接调用无权限接口。
- 权限变更后重新登录。
- 权限变更后刷新当前页面。
- 删除、导出、授权等高风险接口。
- 不同数据范围用户查询同一个列表。

## 最佳实践

- 权限 code 稳定且集中维护。
- 前端权限负责体验，后端权限负责安全。
- 菜单、按钮、API、数据范围分层设计。
- 动态路由初始化要能处理刷新。
- 高风险操作写审计日志。
- 权限变更要有兼容和回滚方案。
