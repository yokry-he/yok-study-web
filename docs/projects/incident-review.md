# 故障复盘模板

## 适合谁看

适合项目上线后出现过故障、联调事故、缓存事故、权限事故或 AI 输出事故，需要把问题沉淀成经验的人。

复盘的目标不是追责，而是让系统下一次更不容易以同样方式失败。

## 什么时候需要复盘

建议复盘这些情况：

- 线上不可用。
- 大量用户报错。
- 权限越权。
- 数据误删或误改。
- 发布后旧页面缓存。
- 数据库慢查询拖垮接口。
- AI 回答泄漏不该看的资料。
- 同类问题重复出现。

小问题也可以轻量复盘。只要它暴露了流程、监控、测试或文档缺口，就值得记录。

## 复盘原则

- 不追责个人。
- 记录事实，不写猜测。
- 先恢复服务，再分析原因。
- 找系统性原因，不只找直接触发点。
- 每个改进项要有负责人和截止时间。
- 复盘结果要回写文档和检查清单。

Google SRE 的复盘文化强调无责复盘，重点是识别促成事故的因素，而不是指责某个人。

## 复盘结构

```md
# 故障名称

## 摘要

## 影响范围

## 时间线

## 发现方式

## 处理过程

## 根因分析

## 做得好的地方

## 做得不好的地方

## 改进项

## 需要更新的文档
```

## 摘要怎么写

摘要要回答：

- 发生了什么。
- 影响了谁。
- 持续多久。
- 当前是否恢复。
- 最主要的改进方向。

示例：

```text
2026-07-02 10:15 到 10:42，后台用户列表接口因新索引缺失导致慢查询，部分用户访问用户管理页面超时。10:42 通过回滚查询逻辑恢复。后续补充索引评审、慢查询告警和发布前压测检查。
```

## 时间线

时间线只记录事实。

| 时间 | 事件 |
| --- | --- |
| 10:15 | 新版本发布 |
| 10:20 | 用户反馈用户列表加载慢 |
| 10:23 | 监控发现 API P95 升高 |
| 10:30 | 定位到用户列表慢查询 |
| 10:42 | 回滚查询逻辑，接口恢复 |
| 11:10 | 完成复盘初稿 |

不要写：

- “某某没有检查”。
- “应该早点发现”。
- “怀疑是数据库问题”。

这些可以在分析里讨论，但时间线只放可验证事实。

## 根因分析

根因不是一句“代码有 bug”。

可以按层分析：

| 层 | 问题 |
| --- | --- |
| 直接触发 | 新查询上线 |
| 技术原因 | WHERE + ORDER BY 缺少复合索引 |
| 测试原因 | 测试数据量太小 |
| 发布原因 | 发布前没有慢查询检查 |
| 监控原因 | 没有针对接口 P95 的告警 |
| 文档原因 | 交付清单没有索引验证项 |

复盘要找到能被改进的系统环节。

## 改进项

每个改进项必须可执行。

| 改进项 | 负责人 | 截止时间 | 验收方式 |
| --- | --- | --- | --- |
| 用户列表补复合索引 | 后端 | 2026-07-05 | EXPLAIN 使用索引 |
| 发布清单增加慢查询检查 | 前端/后端 | 2026-07-06 | 清单已更新 |
| API P95 超阈值告警 | DevOps | 2026-07-08 | 告警规则上线 |

不要写“加强测试”“提高意识”这种无法验收的动作。

## 常见复盘案例

### 1. 权限越权

现象：

用户没有按钮，但直接请求接口成功。

根因：

- 前端做了按钮隐藏。
- 后端没有接口授权。
- 交付清单没有直接请求接口检查。

改进：

- 后端加 `permissionRequired`。
- 高风险接口增加审计日志。
- 权限测试清单加入“直接请求无权限接口”。

### 2. 旧页面缓存

现象：

发布后部分用户仍然看到旧页面。

根因：

- `index.html` 被 CDN 长缓存。
- 发布后只刷新了 JS，没有刷新 HTML。
- 检查清单没有缓存验证。

改进：

- `index.html` 短缓存。
- 发布流程刷新入口 HTML。
- 上线后用无缓存窗口验证。

### 3. AI 回答越权

现象：

普通用户问到了管理员文档内容。

根因：

- 页面入口做了权限。
- RAG 检索层没有权限过滤。
- 评测集没有越权问题样例。

改进：

- 检索前带用户权限上下文。
- 文档 chunk 保存权限元数据。
- 评测集加入越权样例。

## 复盘输出放哪里

| 输出 | 放置位置 |
| --- | --- |
| 具体故障过程 | 当前复盘文档或团队复盘记录 |
| 通用问题 | [真实项目问题库](/projects/real-world-issues) |
| 预防检查项 | [项目交付检查清单](/projects/delivery-checklist) |
| 技术解释 | 对应技术模块 |
| 路线任务 | [阶段任务清单](/roadmap/phase-tasks) |

复盘结束后必须更新文档，否则同类问题会再次出现。

## 最佳实践

- 复盘关注系统改进，不追责个人。
- 时间线只写事实。
- 根因分析要覆盖技术、测试、发布、监控和文档。
- 改进项必须有负责人、时间和验收方式。
- 复盘结果要回写问题库和检查清单。
- 重复出现的问题要升级为流程或自动化检查。
