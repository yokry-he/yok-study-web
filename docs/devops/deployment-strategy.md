# 发布、回滚与环境治理

## 适合谁看

适合已经能把项目部署上去，但还缺少上线检查、版本管理、回滚策略和环境治理的人。

能上线只是第一步。成熟一点的发布流程要保证：

- 发布前知道要发布什么。
- 发布中知道每一步是否成功。
- 发布后知道线上是否真的生效。
- 出问题时能快速回滚。
- 不同环境配置不会混乱。

## 环境划分

常见环境：

| 环境 | 作用 | 注意点 |
| --- | --- | --- |
| local | 本地开发 | 可以使用 mock 和本地代理 |
| dev | 开发联调 | 变动频繁，数据可重置 |
| test | 测试验证 | 尽量接近生产流程 |
| staging | 预发 | 配置和生产接近，数据隔离 |
| production | 生产 | 严格控制权限和发布 |

不要让本地 `.env` 直接复制到生产。每个环境都应该有明确变量来源和审批方式。

## 版本号

每次发布都应该能追踪到：

- Git commit。
- 构建时间。
- 构建人或触发方式。
- 产物版本。
- 环境变量版本。

可以生成：

```json
{
  "version": "2026-07-01-1530",
  "commit": "abc1234",
  "builtAt": "2026-07-01T15:30:00+08:00"
}
```

发布后访问 `/version.json`，确认线上版本。

## 发布目录结构

推荐不要直接覆盖线上目录：

```text
/var/www/my-app/
├─ releases/
│  ├─ 2026-07-01-1200/
│  └─ 2026-07-01-1530/
└─ current -> releases/2026-07-01-1530
```

Nginx 指向：

```nginx
root /var/www/my-app/current;
```

发布时上传到新目录，验证后切换软链接。回滚时把 `current` 切回上一个版本。

## 发布前检查

| 检查项 | 方法 |
| --- | --- |
| 依赖安装 | `npm ci` |
| 类型和 lint | `npm run typecheck`、`npm run lint` |
| 测试 | `npm run test` |
| 生产构建 | `npm run build` |
| 产物预览 | `npm run preview` 或 HTTP 服务 |
| 环境变量 | 校验必要变量是否存在 |
| 变更说明 | 写清楚影响范围和回滚方式 |

没有测试的项目，至少要有构建和关键路由验证。

## 发布后验证

前端项目至少验证：

1. 首页能打开。
2. 二级路由能直接访问和刷新。
3. 登录流程正常。
4. 关键接口正常。
5. 静态资源没有 404。
6. 控制台没有首屏错误。
7. 版本号是新版本。

可以用命令做基础检查：

```bash
curl -I https://example.com/
curl -I https://example.com/admin/users
curl -I https://example.com/api/health
```

## 回滚策略

回滚不是“重新发布旧代码”那么简单。你需要明确：

- 静态资源旧版本是否还在。
- 数据库是否有不可逆迁移。
- CDN 是否需要刷新。
- 后端接口是否兼容旧前端。
- 用户正在访问时是否会白屏。

静态前端回滚通常较简单：

```bash
ln -sfn /var/www/my-app/releases/2026-07-01-1200 /var/www/my-app/current
systemctl reload nginx
```

如果涉及数据库迁移，必须提前设计回滚或兼容方案。

## 灰度发布

灰度发布不是等待固定时间后自动全量，而是在 1%、5%、25%、50% 每个阶段比较错误率、延迟和业务指标，并允许立即停止或回滚。

每次扩量都要记录版本、流量比例、观察窗口、基线差异和决策人，确保发布过程可复盘。

灰度发布是让部分用户先使用新版本，确认稳定后再扩大范围。

常见方式：

- 按用户 ID。
- 按租户。
- 按地区。
- 按流量比例。
- 按内部账号。

前端灰度要注意静态资源版本一致性。不要让同一个页面同时加载新旧不兼容资源。

## 环境变量治理

前端变量：

- `VITE_API_BASE_URL`
- `VITE_APP_ENV`
- `VITE_PUBLIC_PATH`

这些变量构建后会进入静态产物。不能放密钥。

后端变量：

- `DATABASE_URL`
- `REDIS_URL`
- `JWT_SECRET`
- `LOG_LEVEL`

后端变量通常运行时读取，可以由容器平台、系统服务或密钥系统注入。

## 实际项目问题

### 问题：测试环境正常，生产接口错环境

**原因**

构建时使用了错误 `.env`，或者 CI 没有区分环境变量。

**解决方案**

- 构建日志输出非敏感环境标识。
- 启动时生成 `/version.json` 包含环境名。
- CI 按环境分 job 和 secrets。

### 问题：上线后需要回滚，但旧资源被删了

**原因**

发布脚本覆盖目录或清理了旧 hash 静态资源。

**解决方案**

- 使用 releases 目录。
- 保留最近 N 个版本。
- CDN 清理策略不要删除仍可能被旧 HTML 引用的资源。

## 最佳实践

- 每次发布都有版本号。
- 发布目录使用 releases + current。
- 发布后做自动健康检查。
- 回滚方案要在上线前写清楚。
- 数据库迁移优先做向前兼容。
- 环境变量要分类管理，前端不保存密钥。
