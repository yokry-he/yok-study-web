# 云服务与对象存储部署

## 适合谁看

适合已经会用 Nginx、Docker 或 CI/CD，但面对云主机、对象存储、CDN、云数据库、Serverless 时不知道怎么选的人：

- 前端静态资源应该放服务器还是对象存储。
- API 应该部署到云主机、容器平台还是 Serverless。
- 图片和附件不知道该放哪里。
- CDN 缓存导致上线后用户还看到旧版本。
- 云上服务很多，不知道哪些是必要的。

云部署的重点不是“用了多少云产品”，而是根据项目规模选择简单、稳定、可恢复、成本可控的方案。

## 常见部署形态

| 形态 | 适合 | 特点 |
| --- | --- | --- |
| 云主机 | 小团队、后台系统、学习项目 | 可控性强，运维责任也多 |
| 静态托管 | 官网、文档站、前端 SPA | 简单、便宜、适合 CDN |
| 对象存储 + CDN | 静态资源、图片、附件 | 适合大文件和高访问量 |
| 容器服务 | 多服务、需要弹性和隔离 | 适合已有容器化项目 |
| Serverless | 事件驱动、低频 API、轻量任务 | 运维少，但冷启动和限制要评估 |
| 云数据库 | 生产关系型数据 | 备份、高可用、监控更成熟 |

初学项目不要一开始就堆很多云产品。先把路径、环境变量、日志、备份和回滚做好。

## 前端静态站推荐方案

Vue、React、VitePress 这类构建后是静态文件的项目，可以用：

```text
Git push
↓
CI 构建
↓
上传 dist 到对象存储或静态托管
↓
CDN 分发
↓
域名访问
```

关键点：

- 构建产物带 hash。
- `index.html` 缓存时间短。
- JS/CSS 静态资源可以长缓存。
- 发布后清理或刷新 CDN。
- 如果部署到子路径，要统一 Vite base、Router base 和 CDN 路径。

## 对象存储适合放什么

适合：

- 图片。
- 附件。
- 用户上传文件。
- 静态资源。
- 导出文件。
- 备份文件。

不适合：

- 需要频繁小范围修改的结构化业务数据。
- 需要复杂查询的数据。
- 密码、Token 等高敏感明文数据。

对象存储里的文件也需要权限控制。不要把所有 bucket 都设成公开读。

## 文件上传基本链路

推荐后端签名上传：

```text
前端请求上传凭证
↓
后端校验登录和权限
↓
后端生成临时上传地址
↓
前端直传对象存储
↓
前端通知后端保存文件记录
↓
后端返回业务文件 ID
```

好处：

- 文件不必经过应用服务器中转。
- 后端仍然控制权限。
- 上传凭证可以短期有效。
- 数据库保存文件和业务的关系。

数据库里至少保存：

- 文件 ID。
- 对象 key。
- 原始文件名。
- MIME 类型。
- 大小。
- 上传人。
- 业务归属。
- 创建时间。
- 删除状态。

## CDN 缓存策略

常见策略：

| 文件 | 缓存 |
| --- | --- |
| `index.html` | 短缓存或不强缓存 |
| 带 hash 的 JS/CSS | 长缓存 |
| 图片资源 | 根据更新频率设置 |
| API 响应 | 谨慎缓存，必须区分用户和权限 |

前端上线后仍看到旧页面，常见原因是：

- `index.html` 被 CDN 长缓存。
- service worker 缓存了旧资源。
- 子路径 base 配置错误。
- CDN 刷新只刷新了部分路径。

## API 部署怎么选

| 方案 | 适合 |
| --- | --- |
| 云主机 + systemd/PM2 | 小型 Node API、后台系统 |
| Docker Compose | 多服务小项目 |
| 容器服务 / Kubernetes | 多实例、弹性、复杂服务 |
| Serverless Function | 轻量 API、事件任务、低频流量 |

选择标准：

- 团队是否会运维。
- 是否需要自动扩缩容。
- 是否有长连接。
- 是否有后台任务。
- 是否有稳定数据库连接需求。
- 是否需要快速回滚。

不是所有 API 都适合 Serverless。长连接、稳定连接池、长耗时任务要特别评估。

## 云数据库

生产项目优先考虑托管数据库，而不是自己在云主机上手动装数据库。

托管数据库通常提供：

- 自动备份。
- 时间点恢复。
- 监控告警。
- 高可用。
- 版本升级能力。
- 访问控制。

但托管不代表不用管理。仍然要关注：

- 连接数。
- 慢查询。
- 存储空间。
- 备份保留时间。
- 权限账号。
- 费用。

## 域名和 HTTPS

上线必须处理：

- 域名解析。
- HTTPS 证书。
- HTTP 跳转 HTTPS。
- 证书自动续期。
- HSTS 是否开启。
- www 和根域名是否统一。

证书过期是很常见的线上事故。不要依赖手动续期。

## 成本意识

云服务不是越多越好。

容易产生费用的地方：

- CDN 流量。
- 对象存储出网。
- 日志存储。
- 监控指标。
- 云数据库规格。
- 快照和备份。
- 跨区域流量。

建议给每个环境打标签：

- `env=prod`
- `env=staging`
- `owner=team-name`
- `project=xxx`

这样后续查成本、清资源、做账单分析更容易。

## 实际项目问题

### 1. 发布后部分用户还是旧版本

**原因**

`index.html` 或 service worker 缓存没处理好。

**解决方案**

- `index.html` 短缓存。
- 静态资源文件名带 hash。
- 发布后刷新 CDN 入口 HTML。
- PWA 项目设计更新提示。

### 2. 用户上传文件后任何人都能访问

**原因**

bucket 设置了公开读，文件 URL 没有权限控制。

**解决方案**

- 私有 bucket。
- 使用临时签名 URL。
- 后端校验业务权限。
- 下载和导出行为写审计日志。

### 3. 云数据库连接数打满

**原因**

每个实例连接池过大，扩容后总连接数超过数据库上限。

**解决方案**

- 计算实例数乘以每实例连接池大小。
- 设置合理连接池。
- 使用连接池代理或云厂商连接管理能力。
- 排查连接泄漏和慢查询。

### 4. 测试环境长期忘记关闭

**原因**

云资源没有标签、没有所有者、没有生命周期策略。

**解决方案**

- 所有资源打标签。
- 非生产环境自动停机或降配。
- 定期清理未使用资源。
- 账单按项目归属。

## 最佳实践

- 静态站优先考虑静态托管、对象存储和 CDN。
- API 部署按团队能力和业务特性选择，不盲目上复杂平台。
- 对象存储默认私有，公开访问必须有明确理由。
- `index.html` 短缓存，带 hash 资源长缓存。
- 云数据库也要关注连接数、备份、慢查询和权限。
- 域名证书必须自动续期。
- 所有云资源要有环境、项目和负责人标签。
- 发布、回滚、备份、成本和安全都要写进项目文档。
