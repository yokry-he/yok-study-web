# 可观测性：日志、指标与链路追踪

## 适合谁看

适合已经能把项目部署上线，但上线后排查问题主要靠“看一下服务器日志”的学习者：

- 用户说慢，但不知道慢在哪一层。
- 接口 500，只看到一行错误。
- 前端、网关、后端、数据库日志无法串起来。
- 没有告警，事故靠用户反馈。
- 发布后不知道新版本是否健康。

可观测性的目标不是“多装几个监控工具”，而是让系统出问题时能快速回答：发生了什么、影响了谁、从什么时候开始、是哪一次发布或哪一层导致的。

## 三类核心信号

指标负责发现异常，日志与 Trace 负责定位具体请求。下面的看板把错误率、P95、数据库连接池与一次慢查询链路放进同一时间窗口。

排障记录应从告警时间、受影响接口和 Request ID 开始，再进入具体日志与 Trace；敏感请求体和 Token 不得进入这些证据。

| 信号 | 解决什么问题 | 示例 |
| --- | --- | --- |
| Logs 日志 | 发生了什么具体事件 | 登录失败、SQL 错误、发布完成 |
| Metrics 指标 | 系统整体状态如何 | QPS、错误率、CPU、内存、P95 延迟 |
| Traces 链路追踪 | 一次请求经过哪些服务 | 前端 -> 网关 -> API -> 数据库 |

只看日志会缺少趋势，只看指标会缺少上下文，只看链路会缺少业务细节。真实项目里三者要能互相关联。

## 日志应该怎么记

推荐结构化日志：

```json
{
  "level": "error",
  "message": "create order failed",
  "requestId": "req_01H...",
  "userId": "10001",
  "route": "POST /api/orders",
  "errorCode": "PAYMENT_TIMEOUT",
  "durationMs": 842,
  "time": "2026-07-02T10:00:00.000Z"
}
```

比起普通字符串，结构化日志更容易搜索、聚合和告警。

日志至少包含：

- 时间。
- 级别。
- 请求 ID。
- 用户或租户 ID。
- 路由或任务名称。
- 错误码。
- 耗时。
- 版本号或环境名。

不要记录密码、Token、Cookie、身份证、完整手机号等敏感信息。

## request id 是排查入口

一次请求应该贯穿同一个 request id。

```text
浏览器请求
↓
Nginx access log
↓
Node API log
↓
数据库慢查询
↓
错误告警
```

如果每一层都记录 request id，就能从用户的一次报错快速定位到后端日志、数据库查询和具体版本。

常见做法：

- 网关生成 `x-request-id`。
- 后端读取已有 request id，没有就生成。
- 日志统一输出 request id。
- 前端错误上报带上 request id。

## 指标看什么

Web 项目至少关注：

| 指标 | 含义 |
| --- | --- |
| QPS / RPS | 请求量 |
| Error rate | 错误率 |
| P95 / P99 latency | 慢请求体验 |
| CPU / Memory | 资源压力 |
| Disk usage | 磁盘空间 |
| Connection count | 数据库和连接池压力 |
| Queue length | 队列积压 |
| Cache hit rate | 缓存命中率 |

不要只看平均响应时间。平均值可能掩盖少数用户的极慢请求。P95、P99 更适合观察尾部延迟。

## 告警不要太吵

告警应该帮助行动，而不是制造噪音。

推荐按影响设置：

| 告警 | 示例 |
| --- | --- |
| 用户影响 | 5xx 错误率持续 5 分钟超过阈值 |
| 容量风险 | 磁盘使用率超过 85% |
| 依赖故障 | 数据库连接失败持续出现 |
| 发布异常 | 新版本发布后错误率明显上升 |
| 安全风险 | 登录失败次数异常、导出行为异常 |

不要为每一条 error 日志都发告警。短时间可自动恢复的小波动可以记录，但不一定需要打断人。

## 链路追踪解决什么

链路追踪把一次请求拆成多个 span。

```text
GET /api/orders
├─ auth middleware 12ms
├─ order service 30ms
├─ query orders 180ms
└─ render response 8ms
```

它能帮助你知道：

- 是哪个服务慢。
- 哪个数据库查询慢。
- 哪个第三方接口慢。
- 请求在哪一层失败。
- 某次发布后哪条链路变慢。

OpenTelemetry 是常见的标准化可观测性方案，可以采集 traces、metrics 和 logs，并导出到不同后端。

## 前端也需要可观测性

前端可观测性常见内容：

- 页面 JS 错误。
- 未捕获 Promise 错误。
- 接口错误。
- 页面加载性能。
- 白屏时间。
- 用户关键行为。
- 静态资源加载失败。

前端错误上报要带：

- 页面 URL。
- 用户环境。
- 版本号。
- request id。
- source map 映射后的错误位置。

source map 上传给错误平台即可，不建议公开暴露到生产静态目录。

## 发布后观察窗口

每次发布后都应该观察一段时间。

建议检查：

1. 新版本实例是否全部启动。
2. 健康检查是否通过。
3. 5xx 是否上升。
4. P95 延迟是否上升。
5. 前端 JS 错误是否上升。
6. 关键业务转化是否异常。
7. 日志是否出现新错误码。

发布不是“CI 成功”就结束。真正结束是新版本在观察窗口内稳定。

## 实际项目问题

### 1. 用户说偶尔很慢，但服务看起来正常

**原因**

只看平均耗时，没有看 P95/P99，也没有链路追踪。

**解决方案**

- 记录接口耗时。
- 建立 P95/P99 指标。
- 慢请求记录 request id。
- 对慢 SQL 和第三方接口单独计时。

### 2. 线上 500 找不到对应日志

**原因**

前端报错、网关日志和后端日志没有共同 ID。

**解决方案**

- 所有响应返回 request id。
- 前端报错时展示或上报 request id。
- 后端所有日志输出 request id。
- Nginx access log 也记录 request id。

### 3. 告警很多但没人看

**原因**

告警没有分级，低价值噪音太多。

**解决方案**

- 只对持续性、用户影响和容量风险发强告警。
- 错误日志进入看板，不一定全部打电话。
- 每条告警都写处理手册。
- 定期清理无行动价值的告警。

### 4. 发布后过了半小时才发现问题

**原因**

没有发布观察指标。

**解决方案**

- CI/CD 发布完成后自动检查健康接口。
- 观察新版本错误率和延迟。
- 关键业务埋点或接口成功率进入看板。
- 异常自动阻断继续发布或触发回滚流程。

## 最佳实践

- 日志结构化，统一字段名。
- request id 贯穿前端、网关、后端和数据库排查。
- 指标看趋势和尾部延迟，不只看平均值。
- 告警必须可行动。
- 发布后必须有观察窗口。
- 前端错误、接口错误和后端日志要能关联。
- 敏感信息不能进入日志、错误平台或监控标签。
