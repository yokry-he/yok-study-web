# 模块联邦与微前端

## 适合谁看

适合已经遇到多团队、多子应用、多个后台系统集成问题的人：

- 一个后台越来越大，多个团队互相阻塞。
- 希望菜单、用户中心、订单中心独立发布。
- 想共享组件库或工具函数，但不想每次都重新发主应用。
- 听过 Module Federation，但不知道适不适合自己的项目。
- 微前端接入后样式、路由、依赖版本互相污染。

模块联邦和微前端是复杂方案。它们解决的是组织协作和独立发布问题，不是普通项目的默认选择。

## 先判断是否真的需要

适合：

- 多团队并行开发。
- 子应用边界清晰。
- 发布节奏不同。
- 主应用只负责壳、导航和权限上下文。
- 子应用可以独立构建和回滚。

不适合：

- 单团队小项目。
- 只是想把页面分目录。
- 业务边界还不清楚。
- 没有统一设计系统。
- 没有发布治理和监控能力。

如果只是代码太多，先考虑路由拆分、Monorepo、组件库和包体积优化。

## 基本模型

```text
host shell
├─ navigation
├─ auth context
├─ design system
├─ remote: user app
├─ remote: order app
└─ remote: report app
```

| 角色 | 说明 |
| --- | --- |
| host | 主应用，负责加载 remote |
| remote | 子应用，暴露页面或组件 |
| shared | 共享依赖，例如 Vue、React、组件库 |
| runtime | 运行时加载远程模块 |

Module Federation 的核心是让一个构建可以在运行时消费另一个构建暴露的模块。

## 边界怎么划

好的 remote 边界通常按业务域划分：

- 用户中心。
- 订单中心。
- 商品中心。
- 报表中心。
- 系统设置。

不要按技术层划分：

- 所有表格一个 remote。
- 所有弹窗一个 remote。
- 所有接口一个 remote。

微前端边界应该和团队、权限、发布、路由、数据责任一致。

## 主应用负责什么

主应用建议负责：

- 登录态入口。
- 顶部导航和侧边菜单。
- 全局布局。
- 权限上下文。
- 主题和设计 token。
- 错误兜底。
- remote 加载失败提示。

主应用不应该包含所有业务细节，否则 remote 只是形式上的拆分。

## 子应用负责什么

子应用建议负责：

- 自己的路由片段。
- 自己的页面和状态。
- 自己的接口调用。
- 自己的权限点声明。
- 自己的构建和发布。
- 自己的错误监控。

子应用对外暴露稳定入口，不把内部状态随意暴露给主应用。

## 共享依赖要谨慎

常见共享：

- Vue 或 React。
- 路由基础库。
- 状态库。
- 组件库。
- 工具库。

风险：

- 版本不一致。
- 单例依赖冲突。
- 组件库样式污染。
- 主应用升级导致 remote 运行异常。

原则：

- 共享少量稳定依赖。
- 版本范围严格。
- 组件库通过设计系统治理。
- 重大依赖升级需要联调。

## 路由和权限

主应用可以维护全局菜单，但 remote 应该声明自己的路由和权限。

示例：

```ts
export const remoteRoutes = [
  {
    path: '/orders',
    title: '订单管理',
    permission: 'order:read',
    component: () => import('./pages/orders')
  }
]
```

加载流程：

```text
主应用获取用户权限
↓
加载 remote 路由声明
↓
过滤菜单
↓
进入页面时 remote 再做接口权限校验
```

前端菜单隐藏不是安全边界。后端接口仍然必须鉴权。

## 样式隔离

微前端最容易出现样式互相影响。

处理方式：

- 统一设计 token。
- 业务样式使用明确 class。
- 避免全局选择器污染。
- 避免依赖组件库内部 DOM。
- remote 根节点加命名空间。
- 必要时使用 Shadow DOM 或 CSS Modules。

不要在 remote 里写 `body div span` 这类选择器。

## 发布和回滚

微前端发布要能单独回滚。

需要记录：

- host 版本。
- remote 版本。
- remote entry 地址。
- 共享依赖版本。
- 发布时间。
- 发布人。

如果 remote 发布失败，host 应该能：

- 显示错误兜底。
- 回滚 remote 地址。
- 不影响其他 remote。

## 实际项目问题

### 1. remote 加载失败导致整个后台白屏

**原因**

主应用没有 remote 错误边界。

**解决方案**

- 加载 remote 时捕获错误。
- 显示“模块加载失败”状态。
- 提供重试按钮。
- 监控 remote entry 可用性。

### 2. 子应用样式污染主应用

**原因**

remote 使用全局 CSS 和宽泛选择器。

**解决方案**

- remote 根容器命名空间。
- 业务样式明确 class。
- 组件库样式通过主题 token 控制。
- 禁止宽泛后代选择器。

### 3. Vue / React 版本冲突

**原因**

host 和 remote 对共享依赖版本约束不一致。

**解决方案**

- 统一框架版本策略。
- 共享依赖版本进入发布检查。
- 大版本升级建立联调窗口。

### 4. 权限菜单和页面不一致

**原因**

主应用写死菜单，remote 自己又维护权限。

**解决方案**

- remote 声明权限点和路由。
- 主应用聚合菜单。
- 后端接口做最终权限校验。

## 最佳实践

- 微前端先从业务边界和团队边界设计，不从工具开始。
- 主应用负责壳、上下文和兜底，remote 负责业务闭环。
- 共享依赖越少越稳定。
- 样式隔离和设计系统必须提前约定。
- remote 加载失败要有降级。
- 每个 remote 都能独立发布和回滚。
- 接口权限必须在服务端执行。
