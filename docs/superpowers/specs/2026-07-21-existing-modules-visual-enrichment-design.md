# 既有模块视觉讲解完善设计

## 1. 背景

截至 2026-07-21，站点共有 524 篇正式内容文档；另有 1 篇位于 `docs/superpowers` 的 Go 设计规格。327 篇页面包含 Mermaid 图示，但 `docs` 中没有本地 PNG、JPEG、WebP 或 GIF，也没有 Markdown 图片引用。

现有结构图已经能帮助读者理解流程和关系，但以下知识只看文字或抽象图仍然困难：

- HTML 语义、表单、响应式布局和可访问状态需要看到浏览器真实呈现。
- Vue Admin 的列表、表单、详情、权限、审批、消息和数据看板需要看到完整界面及关键状态差异。
- 浏览器 DevTools、网络请求、性能时间线和存储面板需要真实工具界面。
- Docker、CI/CD、日志、监控和故障排查需要看到实际命令结果或仪表盘结构。
- 从零到项目文档需要展示最终成品、关键中间状态和验收结果。

视觉完善的目标是降低理解成本，不是让页面更花哨。所有图片必须承担明确的教学任务，并与文字、代码和可运行示例保持一致。

## 2. 目标

1. 对全部现有模块建立可重复执行的视觉覆盖审计。
2. 为必须观察真实界面或运行结果的章节补充截图、标注图或教学图片。
3. 为图片建立统一组件、替代文本、说明、来源和复现记录。
4. 保留现有 Mermaid 图示，并修正内容过密、移动端不可读或缺少解释的图。
5. 通过自动检查和桌面、移动端浏览器验证防止破图、超大资源和布局溢出。

## 3. 非目标

- 不要求 524 篇正式内容文档每篇都放一张图片。
- 不使用与正文无关的库存照片、装饰插画或技术感背景图。
- 不用生成图片替代必须精确的代码、命令、字段名、状态码和配置。
- 不抓取来源不明、许可证不清晰或禁止再分发的网络图片。
- 不一次性重做站点主题或现有 Mermaid 渲染系统。

## 4. 视觉媒介决策

按以下优先级选择最可靠的表达方式：

| 内容 | 首选媒介 | 原因 |
| --- | --- | --- |
| 流程、状态机、依赖、时序、数据关系 | Mermaid | 准确、可维护、可搜索、适配主题 |
| CSS 布局、组件状态、响应式差异 | 项目代码渲染后的真实截图 | 与实际浏览器行为一致 |
| Vue Admin、React、项目成品 | 本地示例或文档站页面截图 | 可以用步骤复现和回归 |
| DevTools、终端、接口和监控结果 | 真实工具截图或经过脱敏的复现截图 | 展示读者实际会看到的证据 |
| 难以截图的整体心智模型 | `imagegen` 教学图片 | 用空间和场景降低抽象概念门槛 |
| 官方产品特定界面或规范 | 官方许可图片或自行复现 | 保证版本和来源可靠 |

`imagegen` 使用 `scientific-educational` 或 `ui-mockup` 类型。生成图尽量不包含文字；必要标签放在文档标题、图注或可维护的 HTML 标注层中，避免图片内中文错误。生成图不得伪装成真实产品截图。

## 5. 覆盖优先级

### 5.1 P0：真实界面直接决定理解

- `docs/frontend/`：HTML 语义、表单、图片、无障碍和从零项目。
- `docs/css/`：盒模型、Flex、Grid、响应式、动画和项目落地。
- `docs/vue/`：Vue 核心及 Vue Admin 列表、表单、详情、工作台、审批、通知、权限和文件流程。
- `docs/browser/`：请求、缓存、存储、渲染、性能和自动化调试。
- 已有可运行示例对应的 JavaScript、React、Node.js、Java 和 Go 项目页。

### 5.2 P1：运行结果和排障证据重要

- `docs/javascript/`、`docs/typescript/`、`docs/react/`、`docs/meta-frameworks/`。
- `docs/node/`、`docs/java/`、`docs/go/`、`docs/database/`。
- `docs/engineering/`、`docs/devops/`、`docs/ai-engineering/`。
- 对应真实问题库和专项练习。

### 5.3 P2：大规模案例库

- `docs/projects/` 中除专项问题库外的业务案例。
- 学习路线、速查表、贡献文档中确实需要视觉说明的页面。

P2 不按文件名顺序机械处理。优先处理有独特业务状态、复杂交互或高复用架构的案例；高度相似的连续案例共用可维护的领域图解规范，不复制同一张图片。

## 6. 页面审计规则

每篇文档记录以下结果之一：

- `diagram-sufficient`：现有 Mermaid 和文字已经足够，不需要位图。
- `needs-live-screenshot`：需要真实运行截图。
- `needs-annotated-screenshot`：截图需要编号或高亮区域。
- `needs-generated-visual`：抽象场景适合生成教学图片。
- `needs-official-source`：必须展示特定工具或产品官方界面。
- `needs-mermaid-refactor`：已有图过密、错误或移动端不可读。

审计清单至少包含：页面路由、学习难点、当前图示数、建议媒介、图片用途、生成或复现方式、优先级、状态和验收结果。

## 7. 文件与组件

新增以下结构：

```text
docs/
├─ public/images/
│  ├─ frontend/
│  ├─ css/
│  ├─ vue/
│  ├─ browser/
│  ├─ javascript/
│  ├─ react/
│  ├─ node/
│  ├─ java/
│  ├─ go/
│  ├─ database/
│  ├─ engineering/
│  ├─ devops/
│  └─ ai-engineering/
├─ .vitepress/theme/components/DocFigure.vue
└─ contribute/
   ├─ visual-audit.md
   └─ visual-asset-register.md
scripts/
└─ check-visual-assets.mjs
```

`DocFigure.vue` 负责：

- 原生 `figure`、`img`、`figcaption` 语义。
- 必填 `src`、`alt` 和 `caption`。
- 可选 `sourceUrl`、`sourceLabel`、`width`、`height` 和 `zoomable`。
- 图片加载失败状态和可访问的放大查看。
- 固定宽高比或显式尺寸，避免图片加载造成页面跳动。
- 移动端只在图片容器内部滚动或缩放，不产生页面级横向滚动。

组件不在图片上绘制关键文字，不嵌套卡片，也不替代正文解释。

## 8. 资产规范

- 文件名使用小写英文和连字符，例如 `flex-main-cross-axis.png`。
- 路径按技术模块和页面归档，不使用哈希文件名作为源文件名。
- 截图默认使用 2x 像素密度，并在组件中声明逻辑宽高。
- 单张普通文档图片目标不超过 500 KB；确需高分辨率的复杂界面不超过 1.5 MB，并在登记表说明原因。
- 图片不得包含真实令牌、邮箱、连接串、用户数据或本机绝对路径。
- 深色界面截图必须保证正文中的边界和文字仍可读；不能依赖图片说明颜色含义，图注同时写出状态。
- 替代文本描述图中与学习目标有关的信息，不重复“图片展示了”。

## 9. 来源和生成记录

`visual-asset-register.md` 每个资产记录：

- 本地路径和使用页面。
- 教学目的和替代文本。
- 类型：真实截图、生成图片、官方图片或许可图片。
- 截图复现命令、URL、视口和必要数据状态。
- 生成图片的最终 prompt、`imagegen` built-in 模式和生成日期。
- 外部图片的原始 URL、作者或机构、许可证、访问日期和本地修改。

无法确认再使用许可的网络图片不进入仓库。官方文档只提供事实参考但未明确授权图片时，优先自行复现界面或重画结构图。

## 10. 自动检查

`scripts/check-visual-assets.mjs` 检查：

1. `DocFigure` 的 `src` 指向 `docs/public` 下真实文件。
2. `alt` 和 `caption` 非空。
3. 图片扩展名、字节大小和文件名符合规范。
4. 外部来源图片在资产登记中有 URL 和许可证。
5. 生成图片在资产登记中有 prompt 和生成方式。
6. 没有 Markdown 裸图片绕过 `DocFigure` 规范。
7. 资产登记中的本地文件存在，且没有未登记的孤立图片。

该检查接入 `npm run docs:check`，但审计状态不作为构建阻塞条件；审计按 P0、P1、P2 分批推进。

## 11. 验收标准

每个视觉完善批次都必须：

- 完成该批次页面的审计记录。
- 图片来源和复现信息完整。
- `npm run docs:check`、`npm run docs:build` 和 `git diff --check` 通过。
- 在 `1440x900` 和 `390x844` 验证图片加载、图注、放大、焦点、关闭和页面宽度。
- 检查图片请求无 404，控制台无新增错误。
- 对真实截图核对代码版本、数据状态和正文步骤。
- 对生成图片人工核对技术方向，不能把视觉隐喻写成运行事实。

## 12. 分批交付

1. 基础设施批次：组件、样式、资产登记和检查脚本。
2. P0 前端基础与 CSS：真实渲染结果和响应式对比。
3. P0 Vue 与 Vue Admin：核心页面、状态和业务流程截图。
4. P0 浏览器与可运行项目：DevTools、接口结果和最终成品。
5. P1 技术模块：按每个模块最难理解的页面补图。
6. P2 案例库：按独特业务状态和访问价值持续补充。

每批独立构建、浏览器验收和交付，不等待全部 524 篇正式内容审计完成才首次发布。

## 13. 回滚

- 图片与正文引用同批修改；回滚图片时同时移除对应 `DocFigure`。
- 组件和自动检查独立于 Mermaid 渲染，不改变现有 Mermaid 行为。
- 外部图片许可发生变化时可按登记表快速定位并替换。
- 生成图片如果出现技术误导，先移除图片，正文和代码仍保持完整可学习。
