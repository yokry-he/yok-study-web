# 质量检查

## 为什么需要自动检查

文档数量增加后，人工检查很容易漏掉：

- 内部链接指向不存在页面。
- 新增模块没有导览页。
- 技术库有入口，但侧边栏没有配置。
- 核心文章缺少“适合谁看”或“下一步学习”。
- 问题库没有按统一格式沉淀。

自动检查不是替代写作判断，而是先挡住低级结构问题。

## 本地检查命令

```bash
npm run docs:check
```

它会检查：

| 检查项 | 目的 |
| --- | --- |
| 内部链接 | 避免点击 404 |
| 模块导览页 | 保证模块有学习入口 |
| 问题库 | 保证模块能沉淀真实问题 |
| 必备章节 | 保证核心学习页不空洞 |
| 侧边栏路由 | 保证配置里的页面真实存在 |
| 技术库入口 | 保证技术库卡片链接真实存在 |

## 构建检查

```bash
npm run docs:build
```

构建通过说明：

- VitePress 配置语法正确。
- Markdown 可以被渲染。
- 主题组件可以正常编译。
- 站点可以生成生产产物。

构建通过不等于内容质量合格，所以还需要 `docs:check`。

## 搜索索引检查

当前站点使用 VitePress 默认主题的本地搜索。文档规模变大后，搜索索引会成为较大的前端 chunk。

治理策略：

- 页面正文仍完整渲染。
- 本地搜索索引只索引说明性内容。
- 代码块不进入搜索索引，避免大段 SQL、Dockerfile、YAML、TypeScript 示例撑大搜索包。
- Markdown 表格不进入搜索索引，避免状态矩阵、对照表和配置表重复放大索引。
- 自定义 Vue 组件参数不进入搜索索引，避免首页卡片数据污染搜索结果。
- 如果后续需要搜索代码示例，可以为特定速查模块单独评估，而不是默认索引全站代码块。

这样做的取舍是：搜索更偏向概念、问题和章节定位，不偏向代码片段全文检索。

## 浏览器检查

修改全局样式、主题组件、导航或大量表格/代码块后，应至少检查：

- 首页。
- 技术库。
- 本次修改的模块导览页。
- 本次修改的最长文档。
- 390px 左右的移动端宽度。

重点看：

- 页面级横向滚动。
- 顶部导航是否换行异常。
- 侧边栏是否可打开。
- 表格和代码块是否在容器内滚动。
- 内容是否被遮挡。

## 内容人工检查

自动检查无法判断文章是否真的讲清楚。人工检查至少看：

- 是否先解释场景，再解释概念。
- 示例是否接近真实项目。
- 是否说明为什么这么做。
- 是否有常见问题和预防方式。
- 是否有下一步学习。
- 是否引用官方资料或可靠来源。

## 常见失败处理

### 内部链接不存在

检查链接是否写错：

```md
[数据库学习导览](/database/introduction)
```

内部链接必须是站点真实路由，不要写成不存在的路径。

### 模块缺少 introduction

新增目录时，先创建：

```text
docs/<module>/introduction.md
```

导览页负责说明模块定位和学习顺序。

### 模块缺少 troubleshooting

如果模块已经超过 4 篇文档，应补：

```text
docs/<module>/troubleshooting.md
```

问题库用于记录真实项目症状、原因、解决方案和预防建议。

## 发布前推荐流程

```bash
npm run docs:check
npm run docs:build
```

然后在浏览器里检查关键页面。
