# 文档治理

## 为什么需要治理

这个文档站已经从单一 Vue 学习站扩展为程序员技术学习站。模块越多，越容易出现这些问题：

- 顶部导航越来越拥挤。
- 同一个概念在多个页面重复且说法不一致。
- 新模块只有几篇文章，缺少问题库和最佳实践。
- 内部链接失效。
- 技术库说“已完成”，但侧边栏没有入口。
- 真实项目问题散落在不同模块里，难以持续沉淀。

文档治理的目标不是增加流程，而是让扩展保持稳定。

## IA 组织原则

当前站点采用任务优先的信息架构：

```text
学习路线
技术库
前端基础
框架
后端与数据
工程化
实战项目
贡献指南
```

设计理由：

- 初学者可以从学习路线进入。
- 有明确目标的人可以从技术库进入。
- 顶部导航按能力域分组，而不是每新增一个技术就加一个顶层入口。
- 模块内部使用侧边栏承载学习顺序。
- 贡献指南负责约束内容质量和扩展方式。

## 顶部导航规则

顶部导航只放稳定能力域：

| 分组 | 放什么 |
| --- | --- |
| 学习路线 | 面向角色或目标的学习路径 |
| 技术库 | 全站模块入口和扩展路线 |
| 前端基础 | HTML、CSS、JavaScript、TypeScript、浏览器 |
| 框架 | Vue、React、Nuxt、Next 等 |
| 后端与数据 | Node.js、数据库、缓存、消息队列 |
| 工程化 | 构建、部署、DevOps、质量保障 |
| 实战项目 | 可落地项目和真实问题库 |

新增技术时，优先放入已有分组。只有当一个方向足够大、用户任务足够独立、已有分组无法承载时，才考虑新增顶层导航。

## 模块生命周期

| 状态 | 含义 | 要求 |
| --- | --- | --- |
| planned | 已规划但未创建 | 出现在扩展路线中 |
| started | 已有第一批内容 | 有导览和侧边栏入口 |
| first-version | 可作为第一版学习模块 | 至少 5 篇核心文档，含问题库 |
| stable | 可作为长期主入口 | 8 到 12 篇核心文档，结构完整，检查通过 |
| maintenance | 进入维护状态 | 只做更新和问题补充 |
| deprecated | 已过时 | 标注替代入口和迁移说明 |

状态必须体现在 [模块状态](/contribute/module-status) 中。

## 模块标准结构

一个长期模块建议包含：

```text
introduction       学习导览
quick-start        快速开始
core-concepts      核心概念
practice           项目实践
best-practices     最佳实践
troubleshooting    常见问题
resources          资料索引
```

复杂模块可以拆成更具体的主题，例如 Vue 的路由、状态、表单、请求、权限，数据库的建模、索引、事务、迁移。

## 内容类型

| 类型 | 目标 | 典型位置 |
| --- | --- | --- |
| 导览 | 帮用户判断是否该学、按什么顺序学 | `introduction.md` |
| 概念 | 把核心机制讲清楚 | 技术模块内部 |
| 实践 | 解释真实项目如何落地 | 技术模块或项目模块 |
| 问题库 | 记录症状、原因、解决和预防 | `troubleshooting.md` |
| 速查 | 提供 API 和命令快速查阅 | `cheatsheets/` |
| 治理 | 约束结构、质量和贡献方式 | `contribute/` |

不要把速查页写成教程，也不要把教程写成 API 清单。

## 命名规则

- 路由使用英文小写和短横线，例如 `/database/query-optimization`。
- 页面标题使用中文用户语言，例如 `索引与查询优化`。
- 导览页统一使用 `introduction.md`。
- 问题库统一使用 `troubleshooting.md`。
- 顶部导航使用短标签，不写长句。
- 同一概念在不同页面使用同一名称，例如“登录态”“权限码”“动态路由”。

## 扩展流程

新增模块时按这个顺序：

1. 在技术库和扩展路线中说明模块定位。
2. 创建 `introduction.md`。
3. 补 4 到 6 篇核心主题。
4. 补 `troubleshooting.md`。
5. 接入侧边栏和顶部导航分组。
6. 运行 `npm run docs:check`。
7. 运行 `npm run docs:build`。
8. 用浏览器检查关键页面的桌面和移动端布局。

## 质量门槛

每个 stable 模块必须满足：

- 有导览页。
- 有问题库。
- 有真实项目示例。
- 有下一步学习。
- 内部链接全部有效。
- 侧边栏入口完整。
- 构建通过。
- 移动端无页面级横向溢出。
