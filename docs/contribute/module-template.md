# 模块模板

## 模块目标

每个技术模块都应该像一个小型学习路径，而不是一组松散文章。用户进入模块后，应该知道：

- 这个技术解决什么问题。
- 学习顺序是什么。
- 学完能做什么。
- 项目中常见问题怎么解决。

## 推荐目录

```text
technology-name/
├─ introduction.md
├─ quick-start.md
├─ core-concepts.md
├─ practice.md
├─ best-practices.md
├─ troubleshooting.md
└─ resources.md
```

复杂模块可以增加：

```text
forms.md
router.md
state.md
performance.md
testing.md
deployment.md
```

## introduction.md

用途：模块首页。

必须包含：

- 技术定位。
- 适合谁学。
- 学习顺序。
- 当前模块章节地图。
- 推荐实践路径。

示例结构：

```md
# 技术名称学习导览

## 适合谁看

## 你会学到什么

## 学习顺序

## 章节地图

## 实战建议
```

## quick-start.md

用途：帮助用户跑通第一个可用示例。

必须包含：

- 环境要求。
- 创建或安装步骤。
- 最小示例。
- 常见启动问题。
- 下一步。

## core-concepts.md

用途：解释技术核心概念。

要求：

- 每个概念先讲场景。
- 再给代码。
- 最后给真实项目注意事项。

## practice.md

用途：讲真实项目如何落地。

适合内容：

- 目录结构。
- 分层设计。
- 数据流。
- 常见业务场景。
- 与其他模块的关系。

## best-practices.md

用途：提供开发和评审清单。

必须包含：

- 推荐做法。
- 不推荐做法。
- 原因。
- 项目检查清单。

## troubleshooting.md

用途：问题库。

统一格式：

```md
## 问题标题

### 症状

### 常见原因

### 解决方案

### 预防建议
```

## resources.md

用途：外部资料。

要求：

- 官方文档优先。
- 标注适合阶段。
- 不堆无筛选链接。

## 模块完成标准

一个模块可以认为达到第一版完成，需要满足：

- 至少有导览、快速开始、核心概念、最佳实践、问题库。
- 每篇有真实项目示例。
- 导航和侧边栏已接入。
- 全站搜索可以搜到。
- `npm run docs:build` 通过。
- 桌面和移动端无明显布局问题。
