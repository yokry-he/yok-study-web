# 内容写作规范

## 目标

这个文档站不是 API 字典，而是帮助程序员真正学会技术并解决项目问题。每篇文章都应该降低理解难度，解释使用场景、原因、取舍和常见问题。

## 单篇文档结构

推荐结构：

```md
# 标题

## 适合谁看

## 你会学到什么

## 核心概念

## 基础示例

## 实战场景

## 实际项目常见问题

## 最佳实践

## 下一步学习
```

不是每篇都必须完全一样，但核心学习文档应尽量包含这些部分。

## 写作原则

### 先讲场景，再讲 API

不要一上来罗列 API。先说明它解决什么问题。

不推荐：

```text
watch 可以监听响应式数据。
```

推荐：

```text
当搜索关键字变化后需要重新请求列表时，可以使用 watch。它适合处理“状态变化后要执行副作用”的场景。
```

### 示例要接近真实项目

优先使用：

- 用户管理。
- 角色权限。
- 表格筛选。
- 登录态。
- 接口请求。
- 构建部署。

少用纯玩具示例。如果使用 `count`，只用于最小概念说明。

### 每篇都要有问题处理

实际项目问题应按格式写：

```text
症状
原因
解决方案
预防建议
```

这样用户遇到问题时可以直接查。

## 代码示例规范

代码示例要满足：

- 能独立理解。
- 变量命名有业务含义。
- TypeScript 类型尽量明确。
- 不省略关键上下文。
- 不写和主题无关的复杂逻辑。

示例：

```ts
interface UserForm {
  username: string
  mobile: string
  enabled: boolean
}

function createDefaultForm(): UserForm {
  return {
    username: '',
    mobile: '',
    enabled: true
  }
}
```

## 难度控制

每篇文章都应该照顾初学者：

- 第一次出现的概念要解释。
- 缩写要写全。
- 不要连续堆多个新概念。
- 每段尽量短。
- 表格用于比较和选择。
- 复杂流程用步骤或文本图表达。

## 最佳实践写法

最佳实践不要只写“应该怎么做”，还要写“为什么”。

例如：

```text
表单默认值推荐用函数创建，因为每次调用都会得到一个新对象，可以避免新增和编辑弹窗共享同一个对象引用。
```

## 反模式写法

反模式应该包含：

- 错误写法。
- 会导致什么问题。
- 推荐写法。

示例：

```ts
// 不推荐：直接修改 props
props.user.username = 'alice'
```

原因：props 是父组件传入的数据，子组件直接修改会造成数据流混乱。

推荐：复制成表单对象，保存成功后通知父组件刷新。

## 链接规范

内部链接使用真实路由：

```md
[Vue 学习导览](/vue/introduction)
```

不要使用当前页面锚点冒充导航入口。

外部资料优先引用官方文档。

## 更新要求

修改以下内容时必须同步更新文档：

- 目录结构。
- 路由和导航。
- 请求封装。
- 权限模型。
- 表单规则。
- 构建部署。
- 组件库约定。
- 数据库结构或迁移。

## 质量检查清单

提交前检查：

- 是否说明适合谁看。
- 是否解释核心概念。
- 是否有真实示例。
- 是否有实际问题和解决方案。
- 是否有下一步学习。
- 是否没有明显过时内容。
- 是否通过 `npm run docs:build`。
