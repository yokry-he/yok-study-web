# RAG 检索增强生成

## 适合谁看

适合想让模型回答公司文档、产品手册、代码规范、知识库或项目资料的人。

RAG 是 Retrieval-Augmented Generation 的缩写，通常翻译为“检索增强生成”。它的核心思想是：先从知识库检索相关内容，再把这些内容交给模型生成答案。

## 为什么需要 RAG

模型本身不一定知道你的私有数据：

- 企业内部制度。
- 产品最新功能。
- 项目代码规范。
- 客户合同条款。
- 私有 API 文档。
- 历史工单和解决方案。

如果直接问模型，它可能不知道，也可能编造。RAG 通过检索把相关资料放进上下文，降低胡编概率。

## 基本流程

```text
文档采集
↓
清洗和切分
↓
向量化
↓
写入向量库
↓
用户提问
↓
检索相关片段
↓
模型基于片段回答
↓
返回答案和引用来源
```

## 文档切分

切分太大：

- 检索不精准。
- 上下文浪费。
- 成本变高。

切分太小：

- 语义不完整。
- 模型缺少上下文。
- 答案容易断章取义。

建议按标题、段落、业务主题切分，而不是机械按固定字符数切。

## 检索结果要带来源

RAG 的答案最好返回引用：

```json
{
  "answer": "请先在系统设置中启用审批流，然后为角色配置审批权限。",
  "sources": [
    {
      "title": "审批流配置手册",
      "section": "角色权限",
      "url": "/docs/approval/roles"
    }
  ]
}
```

没有来源的答案不适合直接用于高风险决策。

## File Search 和向量库

OpenAI 官方文档中，File Search 可以和 Responses API 配合使用，基于 vector stores 检索知识库内容。对于学习者，可以先理解这件事：

```text
vector store 保存文档索引
file search 从索引中找相关片段
Responses API 使用片段生成答案
```

如果系统已经有自己的向量数据库，也可以自己完成检索，再把检索片段传给模型。

## 实际项目问题

### 问题：明明上传了文档，回答仍然不准

**常见原因**

- 文档切分不合理。
- 检索 query 没改写。
- topK 太少或太多。
- 文档版本过旧。
- prompt 没要求基于来源回答。

**解决方案**

- 查看实际检索到的片段。
- 为答案返回引用来源。
- 为“无资料支持”设计拒答策略。
- 定期清理过期文档。

### 问题：模型回答了知识库没有的内容

**原因**

Prompt 没有限制模型只能基于检索内容回答。

**解决方案**

明确规则：

```text
只能根据 <context> 中的资料回答。
如果资料不足，回答“当前资料不足，无法确认”。
不要使用外部知识补全。
```

### 问题：检索结果权限泄露

**原因**

向量库没有按用户权限过滤文档。

**解决方案**

检索前先根据用户、租户、角色过滤可访问文档。不要先全库检索再靠模型决定能不能回答。

## 最佳实践

- 文档入库前做清洗、去重和版本标记。
- 检索结果返回来源。
- RAG 回答允许“不知道”。
- 权限过滤发生在检索层，不交给模型判断。
- 定期评估召回率、准确率和过期文档比例。
- 关键业务知识库要有文档负责人。
