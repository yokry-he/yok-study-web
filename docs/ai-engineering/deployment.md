# 上线、成本与安全

## 适合谁看

适合准备把 AI 功能从 Demo 推到真实用户面前的人。

AI 功能上线要同时关注效果、成本、延迟、安全、权限和可观测性。否则很容易出现“能用但不可控”的系统。

## 上线前检查

| 检查项 | 为什么重要 |
| --- | --- |
| 输入校验 | 防止超长、空输入、恶意输入 |
| 输出校验 | 防止格式错误进入业务流程 |
| 权限校验 | 防止越权访问文档或工具 |
| 成本预算 | 防止异常请求烧掉预算 |
| 超时控制 | 防止接口长时间挂起 |
| 降级策略 | 模型不可用时仍能给用户反馈 |
| 日志追踪 | 出问题后能定位 |
| 评测通过 | 防止 prompt 或模型退化 |

## 成本控制

成本来源：

- 输入 token。
- 输出 token。
- 检索和向量存储。
- 多轮对话历史。
- 工具调用次数。
- 重试次数。

优化方向：

- 裁剪无关上下文。
- 对常见问题做缓存。
- 控制最大输出长度。
- RAG 只传相关片段。
- 多轮对话做摘要。
- 设置用户、租户或功能维度限额。

## 延迟优化

用户能感受到的延迟来自：

```text
业务接口
↓
检索
↓
模型生成
↓
工具调用
↓
结果校验
```

优化方式：

- 能并行的请求并行。
- 使用流式输出改善等待体验。
- 对高频问题缓存结果。
- 减少不必要工具调用。
- 超长任务转异步任务。

## 安全边界

AI 功能常见风险：

- Prompt injection。
- 越权检索。
- 工具误调用。
- 敏感信息泄露。
- 输出被用户误认为事实。
- 自动执行高风险操作。

基本原则：

- 权限由业务系统判断，不交给模型判断。
- 检索前先做权限过滤。
- 写操作需要确认或审批。
- 日志脱敏。
- 高风险答案显示来源和置信边界。

## 监控指标

建议记录：

| 指标 | 说明 |
| --- | --- |
| 调用次数 | 哪些功能最常用 |
| 成功率 | API 是否稳定 |
| 格式失败率 | 输出是否可被程序消费 |
| 平均耗时 | 用户等待体验 |
| P95/P99 延迟 | 高峰体验 |
| 平均成本 | 单次调用成本 |
| 重试率 | prompt 或服务是否不稳 |
| 用户反馈 | 点赞、点踩、人工修改 |

## 实际项目问题

### 问题：AI 功能成本突然升高

**常见原因**

- 用户提交超长输入。
- 多轮历史没有裁剪。
- RAG 传入太多片段。
- 失败后无限重试。
- Agent 循环调用工具。

**解决方案**

- 设置输入长度限制。
- 设置最大输出长度。
- 设置最大重试次数。
- 设置 Agent 最大步骤。
- 按用户和租户限额。

### 问题：回答泄露了用户无权访问的文档

**原因**

检索层没有权限过滤。

**解决方案**

按用户权限过滤文档后再检索，或者为不同租户和权限域建立隔离索引。

### 问题：模型不可用导致核心流程中断

**原因**

AI 能力被设计成强依赖，没有降级。

**解决方案**

区分核心流程和增强能力。摘要、润色、推荐可以失败降级；支付、审批、权限不能依赖模型单点决策。

## 最佳实践

- AI 调用必须有超时和降级。
- 成本按功能、用户、租户统计。
- 权限过滤在模型调用前完成。
- 写操作默认人工确认。
- 日志脱敏并保留 trace id。
- 上线前跑评测，发布后收集反馈。
