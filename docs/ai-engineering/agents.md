# Agent 工作流

## 适合谁看

适合已经能做普通模型调用和 RAG，但想让 AI 执行多步任务、调用工具、处理状态和协作流程的人。

Agent 不是“更强的聊天机器人”。从工程角度看，Agent 是一个能围绕目标调用工具、保存状态、处理多步流程的应用系统。

## 什么时候需要 Agent

不需要 Agent 的场景：

- 单次摘要。
- 单次分类。
- 简单问答。
- 固定模板生成。

可能需要 Agent 的场景：

- 需要多步计划。
- 需要调用多个工具。
- 需要根据中间结果决定下一步。
- 需要人类审批后继续。
- 需要跨多个专业角色协作。

## Agent 基本组成

| 部分 | 作用 |
| --- | --- |
| 指令 | 定义目标、边界和行为 |
| 模型 | 负责理解、推理和生成 |
| 工具 | 查询数据库、调用 API、搜索文档、执行动作 |
| 状态 | 保存任务上下文和中间结果 |
| 守卫 | 控制权限、输入、输出和风险 |
| 追踪 | 记录每一步为什么发生 |

## 工具调用边界

工具不是随便开放给模型。

工具定义应该明确：

- 工具能做什么。
- 参数 schema。
- 需要什么权限。
- 是否只读。
- 是否需要人工确认。
- 失败如何处理。

示例：

```ts
interface SearchOrderToolInput {
  orderNo: string
}

interface SearchOrderToolResult {
  orderNo: string
  status: 'pending' | 'paid' | 'cancelled'
  total: number
}
```

删除、退款、审批这类写操作不要直接自动执行，至少要有确认步骤。

## Agent 和普通工作流

优先使用普通代码工作流：

```text
固定步骤
固定条件
明确状态机
可用普通代码表达
```

考虑 Agent：

```text
步骤不固定
需要根据自然语言目标规划
需要多个工具组合
需要处理开放式信息
```

不要为了使用 Agent 把简单流程复杂化。

## 实际项目问题

### 问题：Agent 自己调用了危险工具

**原因**

工具权限边界不清楚，缺少审批。

**解决方案**

- 工具按只读和写操作分级。
- 写操作默认需要用户确认。
- 后端再次校验用户权限。
- 记录 tool call 日志。

### 问题：Agent 卡在循环里

**原因**

缺少最大步数、超时和终止条件。

**解决方案**

为每次任务设置：

- 最大工具调用次数。
- 最大耗时。
- 最大成本。
- 明确失败状态。

### 问题：Agent 做了很多事，但不知道为什么

**原因**

没有 trace 和步骤日志。

**解决方案**

记录每一步：

- 输入。
- 工具名。
- 参数摘要。
- 工具结果摘要。
- 耗时。
- 是否失败。

## 最佳实践

- 能用普通代码解决的，不先上 Agent。
- 工具权限由业务系统控制，不由模型决定。
- 写操作需要审批或二次确认。
- 设置最大步骤、超时和成本上限。
- 每个 Agent 任务要有 trace。
- Agent 上线前必须有评测场景和人工验收。
