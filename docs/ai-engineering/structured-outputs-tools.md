# 结构化输出与函数调用

## 适合谁看

适合已经能调用模型，但开始遇到这些问题的人：

- 模型返回的 JSON 偶尔格式错。
- 前端需要字段，模型却输出了一段自然语言。
- 想让模型调用业务接口，但不知道怎么控制边界。
- 工具调用成功了，但参数不可信。
- Agent 执行动作前缺少审批。

结构化输出和函数调用是 AI 工程从“聊天 Demo”走向“业务系统”的关键能力。

## 两类能力不要混淆

| 能力 | 解决什么问题 |
| --- | --- |
| 结构化输出 | 让模型回答可以被程序解析 |
| 函数调用 / 工具调用 | 让模型提出要调用哪个业务能力 |

如果只是要模型返回一个分类结果、摘要字段、表单草稿，用结构化输出。

如果要模型查询订单、创建工单、调用搜索、读取数据库，用函数调用。

## 结构化输出适合什么

适合：

- 文本分类。
- 信息抽取。
- 表单预填。
- 工单标签。
- 风险等级判断。
- 生成可编辑草稿。

示例结果：

```json
{
  "category": "billing",
  "priority": "high",
  "summary": "用户反馈发票金额不一致",
  "needHumanReview": true
}
```

程序后续可以按字段处理，而不是从自然语言里猜。

## Schema 设计原则

Schema 要稳定、简单、可验证。

建议：

- 字段名使用业务语义。
- 枚举写清楚。
- 必填字段明确。
- 不要让模型自由创造字段。
- 对数字、日期、布尔值做运行时校验。

示例：

```ts
type TicketClassification = {
  category: 'billing' | 'account' | 'technical' | 'other'
  priority: 'low' | 'medium' | 'high'
  summary: string
  needHumanReview: boolean
}
```

即使使用结构化输出，也要在代码里再校验一次。

## 函数调用适合什么

适合让模型连接系统能力：

- 查询订单。
- 搜索知识库。
- 创建工单。
- 读取日历。
- 查询库存。
- 生成草稿后保存。

工具定义要写清：

- 工具做什么。
- 什么时候用。
- 参数字段。
- 参数限制。
- 返回结果。
- 是否需要用户确认。

## 工具调用不是直接执行

安全流程应该是：

```text
用户输入
↓
模型决定需要工具
↓
系统校验工具名和参数
↓
检查用户权限
↓
必要时请求用户确认
↓
执行工具
↓
记录日志
↓
把结果交回模型或前端
```

不要让模型直接绕过业务权限执行动作。

## 工具参数必须校验

模型给出的参数不可信。

```ts
function validateCreateTicketArgs(args: unknown) {
  if (!isObject(args)) throw new Error('参数格式错误')
  if (typeof args.title !== 'string') throw new Error('标题必填')
  if (!['low', 'medium', 'high'].includes(args.priority)) {
    throw new Error('优先级不合法')
  }
}
```

校验失败时不要继续执行工具。

## 读工具和写工具

建议区分：

| 类型 | 示例 | 风险 |
| --- | --- | --- |
| 读工具 | 搜索文档、查询订单 | 中低 |
| 写工具 | 创建工单、修改状态 | 中高 |
| 高危工具 | 删除数据、退款、授权 | 高 |

高危工具不建议自动执行，必须有人确认和审计。

## 实际项目问题

### 1. JSON 偶尔解析失败

**原因**

只在 prompt 里写“请返回 JSON”，没有结构化输出和校验。

**解决方案**

- 使用结构化输出能力。
- 定义 schema。
- 对结果做运行时校验。
- 失败时重试或转人工。

### 2. 模型调用了错误工具

**原因**

工具描述不清，多个工具职责重叠。

**解决方案**

- 工具名称使用动词 + 对象。
- 描述写清适用和不适用场景。
- 减少一次暴露给模型的工具数量。
- 记录 tool call 日志用于评测。

### 3. 工具参数越权

**原因**

模型传入了用户不可访问的 resourceId。

**解决方案**

- 工具执行前检查用户权限。
- resourceId 必须由服务端校验。
- 不相信模型生成的角色、权限、用户 ID。

### 4. 自动执行造成误操作

**原因**

写工具没有确认步骤。

**解决方案**

- 写操作先生成计划。
- 用户确认后再执行。
- 高危操作需要二次确认或审批。
- 写审计日志。

## 最佳实践

- 能结构化就不要让程序解析自然语言。
- Schema 要简单稳定。
- 工具参数必须校验。
- 工具执行前必须检查业务权限。
- 读工具和写工具分级管理。
- 高危工具必须确认、审计和可回滚。
- 工具调用日志进入评测集。
