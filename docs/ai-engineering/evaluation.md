# 评测与质量保障

## 适合谁看

适合已经做出 AI 功能 Demo，但不知道如何判断“效果是否变好”“上线后会不会退化”的人。

AI 工程不能只靠主观体验。模型、prompt、RAG、工具和业务数据都会变化，没有评测就无法稳定迭代。

## 为什么需要评测

AI 功能常见变化：

- 换模型。
- 改 prompt。
- 改知识库切分。
- 改检索参数。
- 增加工具。
- 调整输出格式。

每次变化都可能让某些场景变好、另一些场景变差。评测用于发现退化。

## 评测集

评测集是一组代表真实需求的样例。

```json
{
  "id": "ticket-001",
  "input": "页面保存时报 500，但刷新后数据又出现了",
  "expected": {
    "category": "bug",
    "priority": "high"
  }
}
```

评测集应该覆盖：

- 常见正常输入。
- 边界输入。
- 恶意输入。
- 信息不足输入。
- 高价值业务场景。
- 历史线上问题。

## 评测维度

| 维度 | 说明 |
| --- | --- |
| 正确性 | 答案是否对 |
| 完整性 | 是否遗漏关键点 |
| 格式 | 是否符合 JSON/schema/Markdown |
| 引用 | RAG 是否给出正确来源 |
| 安全 | 是否泄露、越权或执行危险操作 |
| 成本 | token 和调用次数是否合理 |
| 延迟 | 用户是否能接受 |

## 自动评测和人工评测

自动评测适合：

- 分类。
- JSON 格式。
- 是否包含关键字段。
- 是否引用来源。
- 是否拒答。

人工评测适合：

- 文案质量。
- 复杂推理。
- 用户满意度。
- 模糊业务判断。

两者要结合。不要指望自动指标覆盖所有体验问题。

## 回归测试

每次改 prompt 或模型前后，对同一批样例运行：

```text
baseline result
new result
diff
pass/fail
```

如果新版本在核心场景退化，就不能直接上线。

## 线上反馈闭环

AI 功能上线后，应收集：

- 用户点赞/点踩。
- 用户修改后的最终答案。
- 失败问题。
- 无法回答问题。
- 高成本请求。
- 高延迟请求。

这些反馈可以进入下一轮评测集。

## 实际项目问题

### 问题：Demo 看起来很好，上线后用户说不准

**原因**

Demo 只覆盖理想输入，评测集没有真实问题。

**解决方案**

从历史工单、客服记录、真实用户问题中抽样，建立评测集。

### 问题：换模型后格式经常错

**原因**

只看答案质量，没有把格式作为评测维度。

**解决方案**

增加 schema 校验和格式失败率指标。

### 问题：RAG 回答看似正确但来源不相关

**原因**

只评估最终答案，没有评估检索片段。

**解决方案**

评测时同时记录检索结果，人工抽查 topK 是否相关。

## 最佳实践

- 每个 AI 功能至少有一组评测样例。
- prompt 变更必须跑回归评测。
- RAG 同时评估检索和生成。
- 输出格式必须自动校验。
- 线上失败样例进入评测集。
- 成本和延迟也属于质量指标。
