# 多模态 AI 应用

## 适合谁看

适合已经会文本模型调用，但希望处理图片、语音、文件或截图的人：

- 想让模型看截图并解释问题。
- 想上传图片生成描述或标签。
- 想做语音输入和语音回复。
- 想让 AI 读取 PDF、合同或表格。
- 不知道多模态输入怎么做权限和成本控制。

多模态不是简单地“多传一个文件”。它会带来文件权限、大小限制、延迟、成本、隐私和输出校验问题。

## 常见模态

| 模态 | 场景 |
| --- | --- |
| 文本 | 摘要、分类、问答、生成 |
| 图片 | 截图分析、票据识别、商品图理解 |
| 音频 | 语音输入、会议转写、语音助手 |
| 文件 | PDF、合同、报告、知识库 |
| 视频 | 片段理解、内容审核、教学分析 |

初学项目建议先做文本 + 图片，或者文本 + 文件。语音和视频通常需要更多实时性和成本治理。

## 图片输入

适合：

- UI 截图分析。
- 表单截图识别。
- 商品图片描述。
- 票据初步识别。
- 图片内容审核辅助。

流程：

```text
用户上传图片
↓
后端校验类型和大小
↓
保存或生成临时访问地址
↓
调用视觉模型
↓
结构化输出
↓
人工确认或业务处理
```

图片识别结果不要直接当作事实。票据、合同、身份证等高风险场景必须人工确认。

## 语音输入和输出

常见能力：

- Speech to text。
- Text to speech。
- 实时语音对话。
- 会议摘要。

工程问题：

- 延迟。
- 噪音。
- 断句。
- 说话人区分。
- 敏感信息。
- 成本。

语音助手如果能执行操作，仍然要走工具权限和确认流程。

## 文件理解

文件常见来源：

- PDF。
- Word。
- Excel。
- Markdown。
- 日志。
- 合同。

处理流程：

```text
上传文件
↓
解析文本
↓
切分 chunk
↓
保存元数据和权限
↓
检索
↓
模型回答
↓
返回引用来源
```

文件问答通常和 RAG 结合，不建议把大文件直接全部塞进 prompt。

## 多模态输出要结构化

例如图片审核结果：

```json
{
  "safe": false,
  "riskTypes": ["privacy"],
  "summary": "图片中包含清晰手机号",
  "needHumanReview": true
}
```

这样业务系统才能决定：

- 是否拦截。
- 是否打码。
- 是否转人工。
- 是否写审计。

## 权限和隐私

多模态通常包含更敏感的数据。

必须考虑：

- 上传人是否有权限。
- 文件是否属于当前租户。
- 图片是否包含隐私信息。
- 语音是否需要用户授权。
- 文件是否进入长期存储。
- 结果是否可以被其他用户看到。

不要把用户上传的文件随意放到公开 bucket。

## 实际项目问题

### 1. 截图分析结果不稳定

**原因**

截图内容太复杂，prompt 没明确关注区域。

**解决方案**

- 让用户标注区域。
- 限制任务，例如“只分析控制台错误”。
- 要求结构化输出。
- 高风险结论人工确认。

### 2. 文件问答答非所问

**原因**

文件没有切分、没有检索，直接把内容丢给模型。

**解决方案**

- 建立 RAG 流程。
- chunk 保存标题、章节、权限。
- 回答带引用来源。

### 3. 语音助手误执行操作

**原因**

语音识别错误后直接触发写工具。

**解决方案**

- 写操作前复述并确认。
- 高危操作走审批。
- 记录音频转写、工具参数和审计日志。

### 4. 多模态成本超预算

**原因**

大图、大文件、长音频无控制。

**解决方案**

- 限制文件大小。
- 压缩图片。
- 长文件先切分。
- 记录每次调用成本。
- 对低价值任务使用更轻量流程。

## 最佳实践

- 多模态输入先校验类型、大小和权限。
- 高风险识别结果必须人工确认。
- 文件问答优先 RAG，不直接塞全文。
- 多模态结果尽量结构化。
- 语音写操作必须确认。
- 记录文件、调用、结果和审计日志。
