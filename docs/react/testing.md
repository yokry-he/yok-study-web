# React 测试策略

## 适合谁看

适合希望减少 React 项目回归问题，但不知道从哪里开始写测试的学习者。

React 测试不应该只测试实现细节，而应该测试用户能看到和能操作的行为。

## 测试分层

| 类型 | 测什么 |
| --- | --- |
| 单元测试 | 工具函数、数据转换、权限判断 |
| 组件测试 | props、交互、表单、条件渲染 |
| E2E 测试 | 登录、列表、新增、编辑、删除 |

## 优先测试什么

优先级：

1. 权限判断。
2. 表单校验。
3. 数据转换。
4. 请求错误处理。
5. 核心组件交互。
6. 关键用户流程。

## 纯函数测试

```ts
export function canAccess(permissions: string[], code: string) {
  return permissions.includes(code)
}
```

测试点：

```ts
expect(canAccess(['system:user:create'], 'system:user:create')).toBe(true)
expect(canAccess([], 'system:user:create')).toBe(false)
```

## 组件测试思路

测试 `UserForm`：

| 场景 | 期望 |
| --- | --- |
| 默认渲染 | 输入框为空 |
| 输入用户名 | value 更新 |
| 点击提交 | 调用 onSubmit |
| 校验失败 | 显示错误 |

不要测试内部 DOM 层级和样式 class。

React 官方 `act` 是测试辅助工具，用于在断言前应用待处理的 React 更新。实际项目通常由测试库封装，但理解它有助于理解“交互后再断言”。

## E2E 测试

适合覆盖：

```text
登录
↓
进入用户管理
↓
新增用户
↓
搜索用户
↓
编辑用户
↓
删除用户
```

E2E 不需要覆盖所有边缘情况，但必须覆盖关键业务路径。

## 实际项目常见问题

### 1. 测试一改 UI 就挂

**原因**

测试依赖内部 DOM 和 class。

**解决方案**

用用户视角测试文本、角色、输入和点击结果。

### 2. 测试依赖真实后端不稳定

**解决方案**

组件测试 mock 数据，E2E 使用稳定测试环境和可重置数据。

### 3. 很难写测试

通常说明业务逻辑和 UI 耦合太深。先抽纯函数、hooks 或 service。

## 最佳实践

- 先测关键逻辑，不追求空覆盖率。
- 组件测试关注行为。
- 不依赖组件内部 DOM。
- E2E 覆盖核心路径。
- 难测试时先调整代码边界。
