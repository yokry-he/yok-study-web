# React 最佳实践

## 适合谁看

适合准备用 React 做真实项目，或者需要评审 React 代码质量的学习者。

## 组件实践

### 组件保持纯净

同样的 props 和 state，组件应该返回同样的 UI。不要在渲染过程中执行请求、修改外部变量或直接操作 DOM。

### 页面组织流程

页面组件负责：

- 请求数据。
- 管理页面状态。
- 打开弹窗。
- 处理提交。
- 组织子组件。

展示组件负责：

- 接收 props。
- 渲染 UI。
- 通过回调通知事件。

## 状态实践

React 官方文档建议状态要有清晰结构，避免冗余和重复状态。

推荐：

- 状态放在最近使用它的组件。
- 多个组件共享时状态提升。
- 跨层稳定上下文用 Context。
- 高频复杂全局状态再考虑状态库。

不推荐：

- 所有状态都全局化。
- 派生状态重复保存。
- 表单每个字段都同步到全局 store。

## Effect 实践

React 官方文档反复强调：你可能不需要 Effect。

适合 Effect：

- 订阅。
- 定时器。
- 浏览器 API。
- 第三方库。
- 网络请求。

不适合 Effect：

- 根据 props/state 计算展示值。
- 响应点击事件。
- 同步两个可直接推导的 state。

## 请求实践

- API 函数和组件分开。
- 请求有 loading 和 error。
- 快速搜索要防抖或取消。
- 旧请求不能覆盖新请求。
- 子组件不要重复请求父组件已有数据。

## 样式实践

- 业务 class 明确。
- 不污染组件库。
- 固定尺寸元素不可压缩。
- 移动端单独检查。
- 组件库基础控件优先复用。

## 项目结构

```text
src/
├─ api/
├─ components/
├─ hooks/
├─ layouts/
├─ pages/
├─ router/
├─ services/
├─ stores/
├─ styles/
└─ types/
```

## 代码评审清单

| 检查项 | 说明 |
| --- | --- |
| 组件 | 是否职责清晰 |
| 状态 | 是否放在正确层级 |
| Effect | 是否真的需要 |
| 列表 | key 是否稳定 |
| 请求 | 是否处理 loading/error |
| 表单 | 是否防重复提交 |
| 样式 | 是否无污染和溢出 |
| 测试 | 关键逻辑是否可测 |

## 常见反模式

### 1. Effect 里同步派生状态

```tsx
useEffect(() => {
  setFullName(`${firstName} ${lastName}`)
}, [firstName, lastName])
```

推荐：

```tsx
const fullName = `${firstName} ${lastName}`
```

### 2. index 作为 key

列表会错乱，使用业务 id。

### 3. Hook 条件调用

Hook 必须在顶层调用。

### 4. 组件承担所有职责

请求、状态、表单、弹窗、权限都写在一个组件里，后续很难维护。
