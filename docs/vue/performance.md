# 性能优化

## 适合谁看

适合已经能完成 Vue 页面，但遇到列表卡顿、首屏慢、重复渲染、包体积过大等问题的学习者。

Vue 官方性能指南强调，Vue 对多数常见场景已经足够高效，只有在复杂场景下才需要额外优化。因此第一原则是：先定位问题，再优化。

## 你会学到什么

- 性能问题分为哪几类。
- Vue 项目里哪些写法容易慢。
- 路由懒加载、组件拆分、列表优化怎么做。
- 如何用浏览器和 Vue DevTools 定位问题。
- 实际项目中大表格、重复请求、包体积过大怎么处理。

## 先区分性能问题类型

| 类型 | 表现 | 常见原因 |
| --- | --- | --- |
| 加载慢 | 首屏白屏时间长 | 包体积大、资源慢、接口慢 |
| 交互卡 | 点击、输入延迟 | 组件重渲染多、计算重 |
| 列表卡 | 滚动卡顿 | DOM 数量太多 |
| 构建慢 | build 时间长 | 依赖多、类型检查慢 |

不同问题的解决方案不同，不要一上来就加缓存或重写组件。

## 路由懒加载

后台系统页面多，应该按路由拆包：

```ts
const routes = [
  {
    path: '/system/users',
    component: () => import('@/views/system/users/index.vue')
  }
]
```

这样用户不需要在首屏加载所有页面代码。

## 减少不必要的计算

复杂计算放到 `computed`：

```ts
const filteredUsers = computed(() => {
  return users.value.filter((user) => {
    return user.username.includes(keyword.value)
  })
})
```

不要在模板里写复杂表达式：

```vue
<!-- 不推荐 -->
<div>{{ users.filter((user) => user.enabled).length }}</div>
```

## 列表优化

### 稳定 key

```vue
<UserRow v-for="user in users" :key="user.id" :user="user" />
```

### 分页

后台管理系统优先使用服务端分页，不要一次加载几万条数据。

### 虚拟列表

如果必须展示大量数据，可以使用虚拟列表。是否引入取决于项目组件库和数据规模，不要过早引入。

## v-memo 和 v-once

静态内容：

```vue
<div v-once>
  这块内容不会再变化
</div>
```

列表局部缓存：

```vue
<UserRow
  v-for="user in users"
  :key="user.id"
  v-memo="[user.enabled, user.selected]"
  :user="user"
/>
```

这些优化只在确定渲染开销明显时使用。过早使用会增加理解成本。

## 组件拆分

拆组件可以减少单文件复杂度，但不一定提升性能。真正影响性能的是：

- 是否大量组件频繁更新。
- props 是否稳定。
- 列表数量是否过大。
- 是否有昂贵计算。

组件拆分优先服务可维护性。

## 包体积优化

常见手段：

- 路由懒加载。
- 不引入整个工具库。
- 图标按需导入。
- 图表、富文本、代码编辑器等重组件异步加载。

示例：

```ts
const CodeEditor = defineAsyncComponent(() => import('./CodeEditor.vue'))
```

## 实际项目常见问题

### 1. 输入搜索框时页面卡顿

**原因**

每输入一个字都过滤大量列表或请求接口。

**解决方案**

- 加防抖。
- 服务端分页搜索。
- 复杂过滤放到 computed。
- 大列表使用虚拟滚动。

### 2. 表格加载几千条数据卡顿

**解决方案**

优先改为服务端分页。前端一次渲染几千行 DOM，体验通常不会好。

### 3. 首页包体积过大

**排查**

- 是否首屏引入图表、富文本、编辑器。
- 是否把所有页面同步导入。
- 是否引入了完整图标库。

**解决方案**

使用路由懒加载和异步组件。

### 4. 页面频繁重复请求

**原因**

多个 watch、onMounted、子组件都在请求同一份数据。

**解决方案**

统一数据入口。页面组件负责请求，子组件通过 props 接收。

## 最佳实践

- 先测量，再优化。
- 首屏少加载重依赖。
- 列表优先分页。
- 模板中避免复杂计算。
- 请求入口统一，避免重复请求。
- 优化要保留可读性，不要为了微小收益牺牲维护成本。
