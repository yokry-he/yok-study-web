# 测试策略

## 适合谁看

适合已经能完成 Vue 功能，但想让项目更稳定、减少回归问题的学习者。

Vue 官方测试指南建议根据不同目标选择不同测试方式。对多数业务项目来说，第一阶段不需要追求测试覆盖率数字，而是先覆盖关键逻辑、关键组件和关键用户流程。

## 你会学到什么

- 单元测试、组件测试、端到端测试分别测什么。
- Vue 项目哪些内容最值得测。
- composable、表单、权限、路由如何设计测试点。
- 实际项目中测试难写通常说明什么。

## 测试类型

| 类型 | 测什么 | 例子 |
| --- | --- | --- |
| 单元测试 | 纯函数、数据转换、权限判断 | `can('user:create')` |
| 组件测试 | 组件输入输出和用户交互 | 表单提交、按钮显示 |
| 端到端测试 | 真实用户流程 | 登录后进入用户管理 |

## 什么最值得先测

优先级：

1. 权限判断。
2. 数据转换函数。
3. 表单校验。
4. 请求错误处理。
5. 登录和动态路由流程。
6. 核心业务页面的关键操作。

不建议第一阶段花大量时间测试纯展示组件。

## 纯函数测试

权限判断：

```ts
export function canAccess(permissions: string[], code: string) {
  return permissions.includes(code)
}
```

测试思路：

```ts
expect(canAccess(['system:user:create'], 'system:user:create')).toBe(true)
expect(canAccess([], 'system:user:create')).toBe(false)
```

## Composable 测试

Composable 如果边界清楚，会很好测。

```ts
export function usePagination(defaultPageSize = 10) {
  const pagination = reactive({
    page: 1,
    pageSize: defaultPageSize,
    total: 0
  })

  function resetPage() {
    pagination.page = 1
  }

  return { pagination, resetPage }
}
```

测试点：

- 默认页码是否为 1。
- pageSize 是否使用传入值。
- resetPage 是否重置页码。

## 组件测试

组件测试关注输入输出，不要测试实现细节。

例如 `PermissionButton`：

| 场景 | 期望 |
| --- | --- |
| 有权限 | 按钮显示 |
| 无权限 | 按钮不显示 |
| disabled | 按钮禁用 |
| 点击 | 触发 click |

不要测试组件内部 DOM 层级，尤其不要依赖组件库内部 class。

## E2E 测试

适合覆盖关键用户流程：

```text
打开登录页
输入账号密码
点击登录
进入工作台
打开用户管理
新增用户
搜索刚新增的用户
```

端到端测试数量不需要太多，但应该覆盖最核心路径。

## 让代码更容易测试

如果测试很难写，通常说明代码边界不清：

- 接口请求直接写在组件深处。
- 权限判断散落在模板。
- 数据转换和 UI 强耦合。
- 组件同时负责请求、状态、路由、弹窗。

解决方式：

- 把纯逻辑抽成函数。
- 把可复用状态抽成 composable。
- 页面组件组织流程，子组件负责展示。

## 实际项目常见问题

### 1. 测试一改 UI 就全挂

**原因**

测试依赖了 DOM 结构或样式 class。

**解决方案**

测试用户可见文本、角色、输入值、事件结果，不测试内部实现。

### 2. 接口测试不稳定

**原因**

测试依赖真实后端数据。

**解决方案**

组件和单元测试使用 mock 数据；端到端测试使用稳定测试环境和可重置数据。

### 3. 不知道从哪里开始写测试

**建议**

先从纯函数开始：

- 权限判断。
- 表单校验。
- 数据转换。
- 菜单过滤。

这些测试成本低、收益高。

## 最佳实践

- 测试关键业务，不追求无意义覆盖率。
- 纯函数优先测试。
- 组件测试关注行为，不依赖内部 DOM。
- E2E 覆盖关键用户流程。
- 难测试的代码通常需要先调整边界。
