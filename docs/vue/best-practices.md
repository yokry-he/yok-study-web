# 最佳实践

## 适合谁看

适合已经完成 Vue 基础学习，准备进入真实项目或重构现有项目的学习者。

这一页把 Vue 模块里的关键原则集中起来，作为开发和评审时的检查清单。

## 组件实践

### 页面组件组织流程

页面组件可以知道业务流程：

- 请求列表。
- 打开弹窗。
- 提交表单。
- 刷新数据。
- 处理权限入口。

但页面组件不应该包含所有细节。搜索表单、表格、弹窗表单可以拆成子组件。

### 展示组件保持输入输出清晰

展示组件通过 props 接收数据，通过 emits 通知事件。

```vue
<UserTable
  :users="users"
  :loading="loading"
  @edit="openEdit"
  @remove="removeUser"
/>
```

不要让展示组件直接依赖路由、全局状态和具体接口，除非它本身就是业务组件。

## 状态实践

### 局部状态留在组件内

例如：

- 弹窗是否打开。
- 表格筛选条件。
- 表单输入内容。
- 当前选中的 tab。

这些通常不需要放 Pinia。

### 跨页面状态放 Pinia

例如：

- token。
- 当前用户信息。
- 权限码。
- 菜单。
- 主题偏好。

Pinia 不是所有状态的垃圾桶。放进去的状态要有清理和恢复策略。

## 请求实践

推荐分层：

```text
api       发送请求
service   编排业务流程
store     保存跨页面状态
view      展示和交互
```

页面不要直接拼大量 URL，不要重复处理 token 和通用错误。

## 表单实践

- 表单类型和接口类型分开。
- 默认值用函数创建。
- 新增和编辑都明确初始化。
- 提交前校验和转换。
- 防重复提交。
- 关闭弹窗清理校验状态。

## 路由实践

- 所有导航使用真实路由。
- 路由 `name` 稳定唯一。
- `meta` 只放路由相关信息。
- 动态路由防重复注册。
- 参数变化用 `watch` 重新请求。
- 部署时验证刷新二级路由不 404。

## 样式实践

- 优先使用组件库和项目设计系统。
- 业务样式命中明确 class。
- 不依赖组件库内部 DOM 层级。
- 固定尺寸视觉元素设置稳定宽高和 `flex-shrink: 0`。
- 修改全局样式后检查桌面和移动端。

## 性能实践

- 先测量，再优化。
- 路由懒加载。
- 列表优先分页。
- 模板避免复杂计算。
- 重组件异步加载。
- 避免重复请求。

## 安全实践

Vue 官方安全指南强调，不要把不可信内容作为组件模板。这一点非常重要。

不要这样做：

```ts
createApp({
  template: userProvidedString
})
```

还要注意：

- `v-html` 只能用于可信 HTML。
- 前端隐藏按钮不等于权限安全。
- token 和敏感信息不要随意存储。
- 后端必须校验所有关键操作。

## 项目评审清单

开发完成一个功能后，检查：

| 检查项 | 说明 |
| --- | --- |
| 路由 | 是否是真实路由，刷新是否正常 |
| 组件 | 是否职责清晰，props/emits 是否稳定 |
| 状态 | 是否区分局部状态和全局状态 |
| 请求 | 是否走 API/service 层 |
| 权限 | 前端展示和后端校验是否都有 |
| 表单 | 是否重置、校验、防重复提交 |
| 样式 | 是否无宽泛选择器和横向溢出 |
| 文档 | 是否同步更新说明 |
| 构建 | `npm run build` 是否通过 |

## 常见反模式

### 1. 一个页面写到底

问题：请求、表单、表格、弹窗、权限都混在一个文件。

处理：拆组件、composable、service。

### 2. 所有状态都进 Pinia

问题：旧页面状态残留、清理困难。

处理：局部状态留在组件内。

### 3. 组件库样式硬覆盖

问题：升级组件库或换主题后大量样式失效。

处理：使用 token、props、CSS 变量或明确业务 class。

### 4. 权限只做前端隐藏

问题：用户仍可直接调用接口。

处理：后端必须校验。

### 5. 只写功能不写文档

问题：后续维护者不知道目录、数据流和配置原因。

处理：功能变更同步更新 README 或模块文档。
